# 代码生成策略

> 每次生成新代码前必须阅读本文档，确保代码质量和开发流程规范。

---

## 一、核心原则：严禁同时生成前后端代码

开发新功能时，必须严格按照以下顺序进行，**禁止同时生成前后端代码**。

---

## 二、标准开发流程

```
第 1 步：定义数据契约（API 文档）
  ├── 确定 API 路径（如 POST /api/auth/login）
  ├── 确定请求参数结构（JSON 字段名、类型）
  ├── 确定响应数据结构（JSON 字段名、嵌套层级）
  └── 输出：API 文档 / 接口定义文件

第 2 步：先写前端代码
  ├── 先写 API 层（src/api/xxx.js）
  │   ├── 确定请求路径
  │   └── 确定期望的返回结构
  ├── 再写页面组件（src/views/xxx/index.vue）
  │   ├── 确定如何使用返回数据
  │   └── 确定数据解包方式
  └── 输出：完整的前端代码

第 3 步：再写后端代码
  ├── 根据前端 API 层确定 Controller 路径
  ├── 根据前端期望的数据结构确定返回格式
  ├── 根据前端数据解包方式调整返回嵌套层级
  └── 输出：完整的后端代码

第 4 步：联调验证
  ├── 启动后端服务
  ├── 启动前端服务
  ├── 逐接口验证请求/响应是否匹配
  └── 修复不匹配的问题
```

---

## 三、数据契约检查清单

每个接口开发前，必须逐项确认：

### 3.1 API 路径一致

- 前端 `request.get('/api/dashboard/stats')`
- 后端 `@RequestMapping("/api/dashboard")` + `@GetMapping("/stats")`
- **前端路径必须与后端路径完全一致**

### 3.2 请求参数一致

- 前端发送的字段名、类型与后端 DTO 一致

### 3.3 响应字段一致

- 前端期望的字段名与后端返回的字段名一致（注意大小写）

### 3.4 数据嵌套层级一致

- 后端返回 `{ code, msg, data: { ... } }`
- Axios 响应拦截器 `return data` → 前端收到的是 `{ ... }` 而不是 `{ code, msg, data }`
- 所以前端应该用 `const s = statsRes` 而不是 `const s = statsRes.data`

### 3.5 拦截器/过滤器路径一致

- Spring Security 的 `SecurityConfig.java` + `JwtAuthenticationFilter.java` 中配置的路径白名单与 API 路径匹配

## 四、商用级代码质量标准（强制）

> 以下规则从 35 项商用审计问题中提炼。每一条都是"上线前必须满足"，不可跳过。

### 4.1 安全（违反 = 阻塞上线）

| 规则 | 正确做法 | 禁止做法 |
|------|----------|----------|
| **敏感值环境变量化** | `${DB_PASSWORD}` / `${JWT_SECRET}`，提供 dev 默认值兜底 | 代码中硬编码密码、密钥、Token |
| **密码 BCrypt 哈希** | `new BCryptPasswordEncoder().encode(raw)` | 明文存储、明文比对 `.equals()` |
| **认证接口限流** | RateLimitFilter 或 AOP 限流（如 60秒5次/IP） | 登录接口无任何调用频率限制 |
| **生产关闭 Swagger** | `application-prod.yml` 中 `springdoc.swagger-ui.enabled: false` | 生产环境暴露 `/swagger-ui.html` |
| **JWT 含角色** | `generateToken()` 签名嵌入 role，前端路由守卫校验 | JWT 只存 username，权限靠客户端 |
| **参数化查询** | MyBatis Plus LambdaQueryWrapper / JDBC `?` 占位符 | 字符串拼接 SQL（`"WHERE name='" + name + "'"`  ） |
| **PII 脱敏** | 日志/响应中手机号 `139****1234` | 完整手机号、身份证号明文输出 |

### 4.2 数据完整性与并发安全（违反 = 资金/库存/状态错乱）

> **审查心法：假设同一段代码会同时被两个请求执行。**

#### 4.2.1 原子 SQL（禁止 read-modify-write）

余额、积分、库存、发券计数——任何 `new = old + delta` 必须一条 SQL 完成。

```java
// ❌ 致命：并发覆盖
Member m = mapper.selectById(id);           // 线程A读到 balance=100
m.setBalance(m.getBalance().add(amount));   // 线程B也在同时读 balance=100
mapper.updateById(m);                        // 后提交的覆盖先提交的

// ✅ 商用：原子 SQL + 行数检查
int rows = jdbcTemplate.update(
    "UPDATE member SET balance = balance + ? WHERE id = ?", amount, id);
if (rows == 0) throw new BusinessException(400, "更新失败");

// ✅ 扣减操作额外带下限守卫
int rows = jdbcTemplate.update(
    "UPDATE member SET balance = balance - ? WHERE id = ? AND balance >= ?", amount, id, amount);
if (rows == 0) throw new BusinessException(400, "余额不足");
```

**检测方法：** grep `setBalance`、`setPoints`、`setStockQty`、`setUsedCount`、`setIssuedQty` — 后面跟 `updateById` 而非 `jdbcTemplate.update("UPDATE ... SET x = x + ?")` 即违规。

#### 4.2.2 状态转换加旧状态守卫（禁止 check-then-act）

```java
// ❌ 两个请求同时通过 if 检查，都执行成功
if (entity.getStatus() == 1) {
    entity.setStatus(2);
    updateById(entity);
}

// ✅ 原子状态转换
int rows = jdbcTemplate.update(
    "UPDATE t SET status = 2 WHERE id = ? AND status = 1", id);
if (rows == 0) throw new BusinessException(400, "状态已变更，请刷新重试");
```

#### 4.2.3 jdbcTemplate.update 必须检查受影响行数

```java
// ❌ 静默失败：WHERE 条件不匹配也不报错
jdbcTemplate.update("UPDATE ... WHERE id = ?", id);

// ✅ 验证成功
int rows = jdbcTemplate.update("UPDATE ... WHERE id = ?", id);
if (rows == 0) throw new BusinessException(500, "操作失败：记录不存在或状态已变更");
```

例外：非关键辅助 UPDATE 允许静默失败，但必须加注释 `// 允许静默失败：不影响主流程一致性`。

#### 4.2.4 jdbcTemplate 写后禁止用 MyBatis 读同一实体

`jdbcTemplate.update()` 绕过 MyBatis 一级缓存，后续 `mapper.selectById()` 返回旧值。

```java
// ❌ 读到旧版本
jdbcTemplate.update("UPDATE consumption_order SET status = 2 WHERE id = ?", id);
ConsumptionOrder refreshed = baseMapper.selectById(id);  // status 可能仍是 1

// ✅ 手动同步内存对象
jdbcTemplate.update("UPDATE consumption_order SET status = 2 WHERE id = ?", id);
order.setStatus(2);
return order;
```

#### 4.2.5 其他数据完整性规则

| 规则 | 正确做法 | 禁止做法 |
|------|----------|----------|
| **金额用 BigDecimal** | `BigDecimal` + `setScale(2, RoundingMode.HALF_UP)` | `double`/`float` 或 `intValue()` 直接截断 |
| **空值防御** | 外部输入（请求头、请求体、DB 查询结果）判空后再用 | `auth.substring(7)` 不判 null |
| **Integer/Long 比较防 NPE** | `Integer.valueOf(1).equals(entity.getStatus())` | `entity.getStatus() == 1`（自动拆箱 NPE） |

#### 4.2.6 数据库连接池配置（生产必配）

> HikariCP 默认值（maxPoolSize=10, connectionTimeout=30s）不满足生产并发需求。必须在 `application.yml` 显式配置。

```yaml
# application.yml — 开发环境
spring:
  datasource:
    hikari:
      minimum-idle: 5
      maximum-pool-size: 20
      connection-timeout: 5000
      idle-timeout: 300000
      max-lifetime: 600000
      connection-test-query: SELECT 1
      pool-name: SalonHikariPool

# application-prod.yml — 生产环境（更大池）
spring:
  datasource:
    hikari:
      minimum-idle: 10
      maximum-pool-size: 50
```

| 参数 | 说明 | 建议值 |
|------|------|--------|
| `maximum-pool-size` | 最大连接数 | 开发 20，生产 50（按 DB 最大连接数调整） |
| `connection-timeout` | 等待连接超时 | 5000ms（默认 30s 太长，客户端已超时） |
| `minimum-idle` | 最小空闲连接 | 5-10（避免冷连接首次查询慢） |
| `pool-name` | 连接池名称 | 自定义（方便监控识别） |

**检测方法：** 启动日志应有 `HikariPool-1 - Starting...` 和 `HikariPool-1 - Start completed.`

### 4.3 错误处理与事务安全（违反 = 线上白屏/500 或数据不一致）

#### 4.3.1 错误处理

| 规则 | 正确做法 | 禁止做法 |
|------|----------|----------|
| **后端 API 调用 try-catch** | 每个外部调用包裹 try-catch，返回 `Result.error(500, "具体原因")` | 异常直接抛到前端，HTTP 500 无信息 |
| **前端 API 调用 try-catch** | `try { await api.xxx() } catch(e) { ElMessage.error(e.message) }` | `api.xxx().then()` 无 `.catch()` |
| **业务异常用 BusinessException** | `throw new BusinessException(400, "库存不足")` → 前端收到 msg | `throw new RuntimeException("库存不足")` → 前端收到"服务器内部错误" |
| **@Async 不能自调用** | 异步方法抽到独立 Service 类中 | `this.asyncMethod()` — Spring AOP 代理不生效 |
| **@Async 必须用显式线程池** | `@Async("auditExecutor")` + `ThreadPoolTaskExecutor` Bean | `@Async` 无参数 → `SimpleAsyncTaskExecutor`（每任务新建线程，生产禁用） |

**@Async 线程池配置（生产必配）：**

```java
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Bean("auditExecutor")
    public Executor auditExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(2);
        executor.setMaxPoolSize(5);
        executor.setQueueCapacity(200);
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.setThreadNamePrefix("audit-");
        executor.initialize();
        return executor;
    }

    @Override
    public Executor getAsyncExecutor() {
        return auditExecutor();  // 默认 @Async 使用的线程池
    }
}
```

**关键参数：** `CallerRunsPolicy` 在队列满时由调用线程执行，防止丢任务。`threadNamePrefix` 方便排查线程占用。

#### 4.3.2 事务安全

**@Transactional 方法内严禁静默吞异常：**

```java
// ❌ 事务正常提交，数据不一致
@Transactional
public void complete(Long id) {
    updateTimerStatus(id, COMPLETED);      // 这条提交了
    try {
        createOrder(id);                    // 这条失败了
    } catch (Exception e) {
        log.error("订单创建失败", e);       // 只记日志，事务照常提交！
    }
}

// ✅ 异常传播，整体回滚
@Transactional(rollbackFor = Exception.class)
public void complete(Long id) {
    updateTimerStatus(id, COMPLETED);
    createOrder(id);                        // 失败则上面也回滚
}
```

**多表/多行写入必须显式声明 `@Transactional(rollbackFor = Exception.class)`。** 确保 checked exception 也触发回滚。

**检测方法：** 在 `@Transactional` 方法内搜索 `catch (Exception` — 任何不重新抛出异常的 catch 块几乎肯定是 bug，除非有明确的降级/补偿逻辑且注释说明了原因。

#### 4.3.3 优雅关闭（生产必配）

任何生产服务必须在 `application.yml` 启用优雅关闭，确保 kill 信号后正在处理的请求能完成。

```yaml
server:
  shutdown: graceful
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
```

**效果：** `kill -15 PID` 后，Tomcat 停止接受新请求，等待 30 秒让进行中的请求完成后再关闭。超时未完成则强制终止。

### 4.4 API 层双向隔离 — Entity 永不跨越 Controller 边界（商用铁律）

> **这是客满满级商用产品的核心架构规则。Entity 是数据库映射对象，Controller 是 HTTP 适配层——两者之间必须由 DTO/VO 完全隔离。进是 DTO，出是 VO，无一例外。**

#### 4.4.1 双向隔离规则

```
客户端 ◄── VO ──── Controller ──── DTO ──► 客户端
                     │
                     │ BeanUtils.copyProperties / 手动映射
                     │
              ┌──────┴──────┐
              │   Entity    │
              └──────┬──────┘
                     │
              ┌──────┴──────┐
              │  Service     │
              │  Mapper      │
              └─────────────┘
```

| 方向 | 数据类型 | 规则 |
|------|---------|------|
| **客户端 → 后端** | `XxxReqDTO` | `@Valid @RequestBody`，仅含客户端有权修改的字段。排除 `id`、`createTime`、`updateTime` 等系统管理字段。 |
| **后端 → 客户端** | `XxxVO` | 所有 API 返回值（GET list/detail、POST 创建后、PUT 更新后）。VO 字段是 Entity 的安全子集——只暴露客户端需要的字段。 |

#### 4.4.2 Controller 方法签名规范

```java
// ✅ 商用级：Entity 不在 Controller 方法签名中出现
@GetMapping
public Result<List<ServiceCategoryVO>> list() { ... }

@PostMapping
public Result<ServiceCategoryVO> create(@Valid @RequestBody ServiceCategoryReqDTO req) { ... }

@PutMapping("/{id}")
public Result<ServiceCategoryVO> update(@PathVariable Long id, @Valid @RequestBody ServiceCategoryReqDTO req) { ... }

@DeleteMapping("/{id}")
public Result<Void> delete(@PathVariable Long id) { ... }

// ❌ 致命违规：Entity 直接暴露
public Result<List<ServiceCategory>> list() { ... }                    // GET 返回 Entity
public Result<ServiceCategory> create(@RequestBody ServiceCategory e) { ... }  // POST 接受 Entity
public Result<ServiceCategory> update(..., @RequestBody ServiceCategory e) { ... } // PUT 接受+返回 Entity
```

**检测方法：** 遍历所有 Controller 的 `@GetMapping`/`@PostMapping`/`@PutMapping` 返回值和方法参数 → 不能出现 Entity 类型。

#### 4.4.3 DTO 设计规范

```java
@Data
public class ServiceCategoryReqDTO {
    @NotBlank(message = "分类名称不能为空")
    private String name;        // ✅ 客户端可设置

    private Integer sort;       // ✅ 客户端可设置

    // ❌ 禁止出现：id, createTime, updateTime — 系统管理字段
}
```

- DTO 字段 = 客户端有权修改的字段，不多不少
- 必填字段加 `@NotBlank` / `@NotNull`
- 手机号等格式字段加 `@Pattern`
- DTO 不跨模块引用（§5.3）
- 每个模块的 DTO 放在 `dto/` 包下

#### 4.4.4 VO 设计规范

```java
@Data
public class ServiceCategoryVO {
    private Long id;            // 客户端需要引用
    private String name;        // 客户端需要展示
    private Integer sort;       // 客户端需要展示
    // 不暴露 createTime/updateTime — 客户端不需要
}
```

- VO 字段 = 客户端需要展示/使用的字段，不多不少
- VO 是 Entity 的安全子集——默认不暴露，按需添加
- 不暴露敏感字段（密码、内部备注、时间戳除非前端需要）
- 每个模块的 VO 放在 `vo/` 包下

#### 4.4.5 Entity ↔ DTO/VO 映射策略

```java
// 方案 1：BeanUtils.copyProperties（项目惯例，TagRuleController 模式）
// 适合 DTO/VO 字段与 Entity 高度重合的场景
private ServiceCategoryVO toVO(ServiceCategory entity) {
    ServiceCategoryVO vo = new ServiceCategoryVO();
    BeanUtils.copyProperties(entity, vo);
    return vo;
}

// 方案 2：手动映射
// 适合 DTO/VO 字段与 Entity 差异大、需要计算的场景
private ServiceCategoryVO toVO(ServiceCategory entity) {
    ServiceCategoryVO vo = new ServiceCategoryVO();
    vo.setId(entity.getId());
    vo.setName(entity.getName());
    vo.setSort(entity.getSort());
    return vo;
}
```

- 映射方法放在 Controller 的 `private` 方法中（简单场景）或在 Service 层（复杂场景，参考 MemberController 模式）
- 避免在 Controller 方法体内内联映射代码——抽取为方法，保持可读性
- 如果 Entity 缺少数据库已有字段（如 `commissionType`），**必须先补齐 Entity**，再做 DTO/VO

#### 4.4.6 其他工程化规则

| 规则 | 正确做法 | 禁止做法 |
|------|----------|----------|
| **批量操作** | `jdbcTemplate.batchUpdate()` 或 MyBatis Plus `saveBatch()` | for 循环逐条 INSERT |
| **构造器注入** | `@RequiredArgsConstructor` + `private final XxxMapper mapper;` | `@Autowired` 字段注入 |
| **分页查所有列表** | `Page<Xxx>` + `IPage<Xxx>` 分页返回 | `list()` 全量返回，前端分页 |
| **无死代码** | 删除未使用的 import、类、方法、文件 | 注释掉的代码块、"留着备用"的方法 |

### 4.5 前端规范

| 规则 | 正确做法 | 禁止做法 |
|------|----------|----------|
| **CSS 变量** | `color: var(--primary-color)` | `color: #409eff` |
| **Scoped 样式** | `<style scoped>` | `<style>`（全局污染） |
| **加载态** | `v-loading="loading"` 或骨架屏 | 点击按钮后无反馈 |
| **空状态** | `<el-empty>` 或 EmptyState 组件 | 空白区域 |
| **无硬编码凭证** | 登录页不显示默认账号密码 | `placeholder="admin / admin123"` |

### 4.6 SQL 变更规范

| 规则 | 做法 |
|------|------|
| **幂等建表** | `CREATE TABLE IF NOT EXISTS` |
| **幂等加列** | `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`（或存储过程模拟） |
| **幂等种子数据** | `INSERT IGNORE INTO` |
| **schema.sql 同步** | 每次加表/加列，立即更新 schema.sql |
| **变更记录** | 同步更新 `docs/16` 变更记录表 |

### 4.7 幂等性（违反 = 重复扣款/重复发券）

#### 4.7.1 支付回调/外部通知必须幂等

任何外部回调（微信支付、支付宝、短信回执）处理第一步 = 幂等性检查。

```java
// 支付回调第一步：查 transaction_id 是否已处理
PaymentDetail existing = paymentDetailMapper.selectOne(
    new LambdaQueryWrapper<PaymentDetail>().eq(PaymentDetail::getTransactionId, callback.getTransactionId()));
if (existing != null) {
    log.warn("重复回调已忽略: transactionId={}", callback.getTransactionId());
    return Result.success();  // 返回成功，不重复处理
}
// 继续处理...
```

#### 4.7.2 用户写操作防重复提交

创建订单、充值、发券等资金操作：
- **前端：** 按钮点击后立即 `disabled`，直到收到响应
- **后端：** 考虑 token/sequence 去重机制

### 4.8 资金与业务闭环（违反 = 资损/僵尸数据）

> **审查心法：每一个业务状态都必须能走完完整生命周期。没有任何中间状态允许永久停留。**

#### 4.8.1 资金安全三要素

任何资金流动（充值/消费/退款）必须同时满足：

| # | 要素 | 说明 |
|---|------|------|
| 1 | **原子 SQL 变更余额** | `UPDATE ... SET balance = balance + ? WHERE id = ?`（§4.2.1） |
| 2 | **业务日志** | `log.info("充值成功: memberId={}, amount={}, newBalance={}", id, amount, newBalance)` |
| 3 | **流水记录** | `rechargeRecordMapper.insert(record)` — 在余额变更**之前**写入，确保可追溯 |

#### 4.8.2 扣减操作下限守卫

任何 `SET x = x - ?` 必须 `WHERE x >= ?`：

```sql
-- ❌ 超额扣减
UPDATE product SET stock_qty = stock_qty - ? WHERE id = ?

-- ✅ 下限守卫
UPDATE product SET stock_qty = stock_qty - ? WHERE id = ? AND stock_qty >= ?
```

`rows == 0` 时错误信息需区分"记录不存在"和"余额/库存不足"。

#### 4.8.3 过期状态处理（杜绝僵尸数据）

所有含 `expire_time` / `expire_date` / `valid_until` 的实体，必须二选一：

| 方案 | 适用场景 | 示例 |
|------|---------|------|
| **定时任务** | 需精确过期（优惠券过期发通知） | `@Scheduled` + `UPDATE ... SET status = 3 WHERE status = 1 AND expire_time < now()` |
| **惰性检查** | 使用时判断即可 | 核销时检查 `if (coupon.getExpireTime().isBefore(now())) → "已过期"` |

**禁止：** 表里有时限字段，但既没有定时任务也没有惰性检查。

#### 4.8.4 派生/冗余字段对账

以下字段会随时间漂移，每个必须有一条对账 SQL：

| 表 | 字段 | 对账 SQL |
|----|------|----------|
| `member` | `total_spent` | `SELECT SUM(total_amount) FROM consumption_order WHERE member_id = ? AND status = 1` |
| `member` | `visit_count` | `SELECT COUNT(*) FROM appointment WHERE member_id = ? AND status >= 2` |
| `coupon_template` | `issued_qty` | `SELECT COUNT(*) FROM coupon WHERE template_id = ?` |

#### 4.8.5 多表事务锁顺序（防死锁）

同一 `@Transactional` 内多表 UPDATE 时，所有调用路径必须按一致顺序访问表：

```
全局锁顺序（按表名字母序）：
appointment → consumption_order → coupon → coupon_template → member → payment_detail
```

#### 4.8.6 外部服务调用超时+降级+熔断

依赖外部系统（支付 SDK、短信 SDK、AI Python）必须有超时和降级：

```java
// ❌ 无超时无降级，AI 挂掉拖死整个请求
String result = aiService.predict(data);

// ✅ 带超时 + 降级
try {
    String result = aiClient.predict(data).orTimeout(5, TimeUnit.SECONDS);
} catch (TimeoutException e) {
    log.warn("AI 服务超时，使用默认推荐");
    result = getDefaultRecommendations();
}
```

**熔断器（生产推荐）：** 连续失败 N 次后快速失败，不每次等待超时。单体阶段降级逻辑手动实现，商用上线后引入 `resilience4j` 或 `Sentinel`。当前至少保证：超时配置 + 降级返回 + 不阻塞主流程。

### 4.9 定时任务安全（违反 = 重复执行/批量中断/多实例冲突）

**E0 — 多实例下定时任务必须获取分布式锁：**

单体部署时没问题，但多实例（≥2）时同一任务会同时触发。所有 `@Scheduled` 方法的第一步必须是获取分布式锁。

```java
// 单体阶段（MySQL GET_LOCK）：
String lockKey = "task_name";
Integer acquired = jdbcTemplate.queryForObject(
    "SELECT GET_LOCK(?, 0)", Integer.class, lockKey);
if (acquired == null || acquired != 1) {
    log.info("锁被其他实例持有，跳过");
    return;
}
try {
    // ... 业务逻辑 ...
} finally {
    jdbcTemplate.execute("SELECT RELEASE_LOCK('" + lockKey + "')");
}

// 生产阶段（Redis Redisson，引入 Redis 后替换）：
RLock lock = redissonClient.getLock("task_name");
if (!lock.tryLock()) { log.info("锁被其他实例持有，跳过"); return; }
try { /* 业务逻辑 */ } finally { lock.unlock(); }
```

**检测方法：** grep `@Scheduled` → 确认方法第一段逻辑是获取锁。

**E1 — 定时任务本身必须是幂等的：**

```java
// ❌ 先查再改，重复执行会重复处理
// ✅ 用一条原子 SQL，WHERE 条件自身就是幂等保证
jdbcTemplate.update(
    "UPDATE coupon SET status = 3 WHERE status = 1 AND expire_time < ?", LocalDateTime.now());
```

**E2 — 批量处理逐条容错：**

```java
// ❌ 一条失败，整批回滚
for (Member m : members) { issueCoupon(m); }

// ✅ 逐条 try-catch
for (Member m : members) {
    try { issueCoupon(m); } catch (Exception e) { log.error("发券失败: memberId={}", m.getId(), e); }
}
```

### 4.10 校验分层与输入安全

> **校验分两层：Controller 做格式校验，Service 做业务规则校验。各司其职，不越界。**

| 层 | 职责 | 工具 | 示例 |
|----|------|------|------|
| **Controller** | 格式校验（非空、长度、正则） | `@Valid` + `@NotBlank` / `@NotNull` / `@Pattern` | "手机号格式不正确" |
| **Service** | 业务规则校验（唯一性、状态机、权限） | `BusinessException` | "该手机号已被注册" |

```java
// Controller — 只做格式校验
@PostMapping
public Result<MemberVO> create(@Valid @RequestBody MemberReqDTO req) {
    return Result.success(memberService.createMember(req));
}

// Service — 做业务规则校验
public Member createMember(MemberReqDTO req) {
    if (memberMapper.existsByPhone(req.getPhone())) {
        throw new BusinessException(400, "该手机号已被注册");
    }
    // 继续创建...
}
```

**输入安全附加规则：**
- 分页参数必须设上限：`@Min(1) int page`，`pageSize` 默认 10，最大 100
- 用户输入的文本内容存储前做 HTML 转义（防 XSS）
- 文件上传必须限制大小（≤10MB）和类型白名单（仅 `.xlsx`、`.jpg`、`.png`）

### 4.11 日志规范（违反 = 线上问题无法排查）

| 级别 | 使用场景 | 示例 |
|------|---------|------|
| **ERROR** | 系统异常，需要立即关注 | 数据库连接失败、第三方支付回调验签失败 |
| **WARN** | 业务异常，可自动恢复 | 登录失败、AI 服务降级、幂等回调被忽略 |
| **INFO** | 关键业务节点 | 订单创建、支付成功、会员注册、充值到账 |
| **DEBUG** | 调试信息，生产关闭 | SQL 参数、方法入参出参 |

**铁律：**
- **禁止打印敏感信息：** 密码、完整手机号、身份证号、Token — 必须脱敏（`138****1234`）
- **资金操作必须 INFO：** `log.info("充值成功: memberId={}, amount={}, newBalance={}", id, amount, newBalance);`
- **异常必须带堆栈：** `log.error("支付失败: orderId={}", orderId, e);` — 第三个参数传异常对象
- **禁止字符串拼接：** `log.info("会员" + name + "登录成功")` → `log.info("会员登录: name={}", name)`
- **禁止 `System.out.println`** 和 `e.printStackTrace()` — 统一用 Slf4j

**结构化日志（生产推荐）：** JSON 格式输出便于 ELK/Loki 采集。通过 `logback-spring.xml` 或 `application.yml` 切换：

```yaml
# application-prod.yml — JSON 格式
logging:
  pattern:
    console: '{"time":"%d{ISO8601}","level":"%level","traceId":"%mdc{traceId}","logger":"%logger","msg":"%msg"}%n'
```

开发环境保留可读格式，生产环境切 JSON。

### 4.12 结构化错误码与请求追踪（商用铁律 — Google AIP-193 对标）

> **客户投诉时说"返回了错误码 MEMBER_001，requestId a1b2c3d4"，运维能精确查到那次请求的完整链路。这就是商用级排障能力。**

#### 4.12.1 错误码体系：业务码 ≠ HTTP 状态码

**现状问题：** `ErrorCode.PHONE_EXISTS.getCode()` 返回 `400`，和 `ErrorCode.BAD_REQUEST` 相同。前端拿到 400 不知道是"手机号已注册"还是"参数格式错误"——无法程序化处理。

**商用标准：** 业务错误码是唯一的字符串标识，HTTP 状态码是传输层的事。二者分离。

```java
// ❌ 当前：业务码就是 HTTP 状态码，无法区分具体错误
BAD_REQUEST(400, "请求参数错误"),
PHONE_EXISTS(400, "手机号已存在"),  // 和上面 code 相同！

// ✅ 商用：业务码唯一，HTTP 状态码独立
BAD_REQUEST("COMMON_001", 400, "请求参数错误"),
PHONE_EXISTS("MEMBER_001", 400, "手机号已存在"),
INSUFFICIENT_BALANCE("PAYMENT_002", 400, "余额不足"),
COUPON_EXPIRED("COUPON_003", 400, "优惠券已过期"),
```

**错误码命名规范：** `{模块}_{序号}`

| 模块 | 前缀 | 示例 |
|------|------|------|
| 通用 | `COMMON_` | `COMMON_001` 参数错误, `COMMON_002` 未授权 |
| 会员 | `MEMBER_` | `MEMBER_001` 手机号已存在, `MEMBER_002` 余额不足 |
| 优惠券 | `COUPON_` | `COUPON_001` 模板不存在, `COUPON_002` 已过期 |
| 订单 | `ORDER_` | `ORDER_001` 订单不存在, `ORDER_002` 已退款 |
| 支付 | `PAYMENT_` | `PAYMENT_001` 余额不足, `PAYMENT_002` 支付失败 |
| 库存 | `STOCK_` | `STOCK_001` 商品不存在, `STOCK_002` 库存不足 |
| 预约 | `APPT_` | `APPT_001` 预约不存在, `APPT_002` 状态不符 |
| AI | `AI_` | `AI_001` 服务不可用, `AI_002` 搜索超时 |

**ErrorCode 枚举结构：**

```java
@Getter
@AllArgsConstructor
public enum ErrorCode {
    // 通用
    BAD_REQUEST("COMMON_001", 400, "请求参数错误"),
    UNAUTHORIZED("COMMON_002", 401, "未登录或Token已过期"),
    FORBIDDEN("COMMON_003", 403, "无权限访问"),
    NOT_FOUND("COMMON_004", 404, "请求的资源不存在"),
    INTERNAL_ERROR("COMMON_005", 500, "服务器内部错误"),

    // 会员
    MEMBER_PHONE_EXISTS("MEMBER_001", 400, "手机号已存在"),
    MEMBER_NOT_FOUND("MEMBER_002", 404, "会员不存在"),
    MEMBER_INSUFFICIENT_BALANCE("MEMBER_003", 400, "余额不足"),

    // 优惠券
    COUPON_NOT_FOUND("COUPON_001", 404, "优惠券不存在"),
    COUPON_ALREADY_USED("COUPON_002", 400, "优惠券已使用"),
    COUPON_EXPIRED("COUPON_003", 400, "优惠券已过期"),
    COUPON_SOLD_OUT("COUPON_004", 400, "优惠券已发完"),

    // ... 按模块扩展

    /** 唯一业务错误码 */
    private final String errorCode;
    /** HTTP 状态码 */
    private final int httpStatus;
    /** 错误消息 */
    private final String message;
}
```

**API 响应格式：**

```json
// ✅ 成功
{ "code": 200, "msg": "操作成功", "data": {...}, "requestId": "a1b2c3d4-e5f6-..." }

// ✅ 失败 — errorCode 唯一标识错误，requestId 用于追踪
{ "code": 400, "msg": "手机号已存在", "errorCode": "MEMBER_001", "requestId": "a1b2c3d4-e5f6-..." }
```

#### 4.12.2 请求追踪 ID（requestId）

每个 HTTP 请求进入时生成唯一 `requestId`（UUID v4），贯穿整个请求生命周期：

```
客户端请求
  │
  ▼
TraceFilter ──► 生成 requestId = UUID.randomUUID()
  │             ├── 放入 MDC（自动出现在所有日志中）
  │             ├── 放入 Response Header（X-Request-Id）
  │             └── 注入 Result.requestId
  ▼
Controller → Service → Mapper
  │            │          │
  │            │          └── SQL 日志自动带 requestId
  │            └── log.info("创建订单: orderId={}, requestId={}", ...)
  └── 返回 Result { ..., requestId: "a1b2c3d4" }
```

**TraceFilter 实现要点：**

```java
@Component
public class TraceFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws IOException, ServletException {
        String requestId = UUID.randomUUID().toString();
        MDC.put("requestId", requestId);
        response.setHeader("X-Request-Id", requestId);
        request.setAttribute("requestId", requestId);
        try {
            chain.doFilter(request, response);
        } finally {
            MDC.remove("requestId");  // 防止线程池复用污染
        }
    }
}
```

**Result 增加 requestId 字段：**

```java
@Data
public class Result<T> {
    private int code;
    private String msg;
    private String errorCode;   // 新增：业务错误码
    private String requestId;   // 新增：请求追踪ID
    private T data;
    // ...
}
```

**logback-spring.xml 中 MDC 变量自动出现在日志：**

```xml
<pattern>%d{HH:mm:ss} [%thread] %-5level %logger{36} - [%X{requestId}] %msg%n</pattern>
<!-- 输出: 14:32:15 [http-nio-8080-1] INFO  c.s.o.OrderService - [a1b2c3d4] 订单创建成功 -->
```

#### 4.12.3 迁移策略

现有代码中有 **40+ 处** `new BusinessException(400, "...")` 使用裸数字。不可一次性重构。

| 阶段 | 操作 | 范围 |
|------|------|------|
| **立即** | 新代码必须用 `new BusinessException(ErrorCode.XXX)` | 所有新 Controller/Service |
| **渐进** | 修改到旧文件时，顺便替换为 ErrorCode | 按文件 |
| **最终** | 全量替换所有裸数字异常 | 全项目 |

**铁律：** 从现在起，`new BusinessException(int, String)` 视为违规。必须 `new BusinessException(ErrorCode.XXX)` 或 `new BusinessException(ErrorCode.XXX, "自定义补充")`。

### 4.14 生产环境基础设施预留模式（商用铁律）

> **核心理念：功能开发期不引入 Redis / RabbitMQ 等中间件，但代码结构必须为将来留好接口。现在是"写代码"，将来是"改配置"。**

#### 4.14.1 为什么不在开发期引入中间件

| 理由 | 说明 |
|------|------|
| **迭代速度** | 每加一个中间件，开发循环多一步确认和排障。`mvn run` + `npm run dev` 10 秒进入状态，应保持 |
| **单实例足够** | 日活 < 100 场景下，单机 MySQL + 内存级限流/锁完全够用 |
| **代码优先** | 先保证代码结构正确（DTO/VO 隔离、原子 SQL、幂等），这些不依赖任何中间件 |
| **Docker Compose** | 需要时一行命令启动中间件，不需要手动安装配置 |

#### 4.14.2 模式 1：缓存预留 — `@Cacheable` + NoOpCacheManager

**现在写（零依赖）：**

```java
// config/CacheConfig.java — 当前：空缓存管理器，不生效但不报错
@Configuration
@EnableCaching
public class CacheConfig {
    @Bean
    public CacheManager cacheManager() {
        return new NoOpCacheManager();  // @Cacheable 注解不缓存，直接执行方法
    }
}

// Service 中直接加 @Cacheable（现在不生效，但代码已就位）
@Override
@Cacheable(value = "services", key = "'all'")
public List<ServiceItem> list() { ... }

@Override
@CacheEvict(value = "services", key = "'all'")
public ServiceItem save(ServiceItem item) { ... }
```

**将来激活：** 引入 `spring-boot-starter-data-redis` → 删 NoOpCacheManager → 换 `RedisCacheManager`。所有 `@Cacheable` 立即生效，零业务代码改动。

**候选缓存：** 服务列表、会员等级、商品分类、员工列表——这些"高频读、低频写"的数据。

#### 4.14.3 模式 2：事件预留 — `ApplicationEvent` + `@EventListener`

**现在写（同步，无 MQ 依赖）：**

```java
// 发布事件
@Transactional(rollbackFor = Exception.class)
public void createOrder(CreateOrderDTO dto) {
    // ... 核心写入 ...
    eventPublisher.publishEvent(new OrderCreatedEvent(order.getId(), memberId, amount));
}

// 监听事件（当前同步执行）
@Component
public class OrderEventListener {
    @EventListener
    @Transactional(rollbackFor = Exception.class)
    public void onOrderCreated(OrderCreatedEvent event) {
        pointsService.addPoints(event.memberId(), calculatePoints(event));
        memberService.checkAndUpgrade(event.memberId());
    }
}
```

**将来激活：** 引入 `spring-boot-starter-amqp` → `eventPublisher.publishEvent()` 改为 `rabbitTemplate.convertAndSend()`，`@EventListener` 改为 `@RabbitListener`。方法体复用。

**适用场景：** 订单创建（发积分 + 升级检查）、支付回调、短信发送、审计日志。

#### 4.14.4 模式 3：配置外置 — `${ENV_VAR}` 区分环境

```yaml

    start_period: 40s
```

- **Liveness:** 容器是否存活（死锁/死循环 → false → 容器重启）
- **Readiness:** 是否准备好接受流量（DB/Redis 不可用 → false → 摘除流量，不重启）

#### 4.14.6 中间件引入三阶段总纲

```
当前（功能开发期）          上线前固化期             生产运维期
───────────────     │     ─────────────     │     ─────────
零成本治理 ✅          │     Redis 引入          │     水平扩展
@Cacheable 预留       │     MQ 异步化           │     多实例
ApplicationEvent 预留  │     可观测性             │     压测调优
配置外置              │     健康检查             │
                      │                        │
   改代码不改架构        改配置+少量代码           改部署拓扑
```

| 阶段 | 何时触发 | 引入内容 |
|------|---------|---------|
| **现在** | 立即 | 连接池配置、@Async 线程池、优雅关闭、健康检查 |
| **每次写新查询** | 惯例 | 顺手加 `@Cacheable`（当前不生效） |
| **每次写新业务** | 惯例 | 发 `ApplicationEvent`（当前同步） |
| **功能全部开发完** | 硬性前提 | Docker Compose 启动 Redis + MQ，集中改造 |
| **部署 ≥ 2 实例** | 硬性前提 | Redis（分布式锁/限流/会话）+ MQ 必须就位 |
| **上线商用** | 硬性前提 | Phase 9-13 全部完成（详见 docs/12 §十） |

#### 4.14.7 关键原则

1. **中间件是放大器，不是替代品。** 如果代码本身有并发 Bug（read-modify-write），加 Redis 只是让 Bug 更快发生
2. **能在代码层解决的问题，不用中间件。** 原子 SQL（`WHERE status=old`）和乐观锁（`rows==0`）能处理的并发，不引入分布式锁
3. **Docker Compose 管理中间件生命周期。** 开发时不启动，需要时一行：`docker compose -f docker-compose.dev.yml up -d`
4. **代码结构先对齐，中间件后接入。** 现在的 `@Cacheable`、`ApplicationEvent`、`${ENV_VAR}` 就是为将来准备好的接口

> 当前为单体 Spring Boot 项目，后续将拆分为 Spring Cloud 微服务架构。
> 开发新功能时必须遵守以下规则，降低未来迁移成本。

### 5.1 模块隔离原则

每个业务模块必须保持独立，禁止跨模块直接调用：

```
✅ 正确：MemberService 只操作 member 相关表
❌ 错误：MemberService 直接调用 ConsumptionOrderMapper 查订单
```

跨模块数据需求通过以下方式解决：

| 场景 | 单体阶段做法 | 微服务阶段做法 |
|------|-------------|---------------|
| 查询会员的消费记录 | MemberService 中注入 OrderMapper（允许临时耦合） | Feign 调用 order-service |
| 扣减会员余额 | OrderService 中注入 MemberMapper（允许临时耦合） | Feign 调用 member-service |
| 查询服务价格 | OrderService 中注入 ServiceItemMapper（允许临时耦合） | Feign 调用 service-service |

> **单体阶段允许的跨模块 Mapper 注入，必须在代码中标注 `@Deprecated(since="微服务迁移后移除")` 注释，方便后续清理。**

### 5.2 跨模块调用走接口抽象

跨模块调用时，先定义接口，再实现。迁移时只需替换实现类，业务代码不用改。

```
单体阶段：
  OrderService
    → 调用 MemberService 接口（不是直接注入 MemberMapper）
    → MemberServiceImpl 是当前单体实现

微服务阶段：
  OrderService
    → 调用 MemberService 接口（接口不变）
    → MemberServiceFeignClient 是远程调用实现
```

具体做法：
- 每个模块的 Service 接口定义放在本模块的 `service/` 包下
- 跨模块调用时，只依赖接口，不依赖实现类
- 单体阶段用 `@Service` 注入本地实现，微服务阶段用 `@FeignClient` 注入远程实现

### 5.3 DTO/VO 不跨模块引用

```
❌ 错误：member 模块引用 consumption 模块的 CreateOrderDTO
✅ 正确：每个模块定义自己的 DTO，即使字段相同
```

原因：拆微服务后，不同服务独立部署，不能共享类。现在养成习惯，每个模块自包含。

### 5.4 ID 生成策略

所有表的主键 ID 用 UUID 或雪花算法，不用数据库自增：

```java
// 实体类中
@TableId(type = IdType.ASSIGN_UUID)  // 或 ASSIGN_ID（雪花算法）
private String id;
```

这样迁移时不需要改 ID 生成逻辑。

### 5.5 数据库设计原则

- 每个模块的表必须有独立的前缀（如 `member_`、`service_`、`order_`）
- 禁止表之间的外键约束，用业务逻辑保证数据一致性
- 跨模块查询只查本模块主表，关联数据通过 API 聚合

### 5.6 API 路径与模块名对齐

前端 API 路径前缀与未来微服务名对应：

| 当前 API 文件 | 路径前缀 | 未来微服务 |
|--------------|---------|-----------|
| `api/member.js` | `/api/member/**` | `member-service` |
| `api/consumption.js` | `/api/consumption/**` | `order-service` |
| `api/service.js` | `/api/service-category` + `/api/service-item` | `service-service` |
| `api/employee.js` | `/api/employee/**` | `employee-service` |
| `api/auth.js` | `/api/auth/**` | `auth-service` |

### 5.7 公共模块抽离

`common/` 包下的内容未来将抽成独立 JAR：

```
common/
├── base/        → BaseEntity（所有实体继承）
├── result/      → Result, PageResult（统一响应）
├── exception/   → BusinessException, ErrorCode, GlobalExceptionHandler
├── util/        → JwtUtil 等工具类
└── security/    → SecurityConfig + JwtAuthenticationFilter（认证过滤器）
```

开发时注意：
- 公共模块**不能依赖任何业务模块**
- 公共模块中的工具类必须是纯静态方法，无状态

### 5.8 配置外置

- 数据库连接、JWT 密钥等配置在 `application.yml`，不硬编码
- 后续迁移到配置中心（Nacos Config）时只需改配置源，代码不动

### 5.9 策略影响总结

以上策略**不会增加额外工作量**，只是编码习惯的调整：

| 策略 | 对开发的影响 |
|------|------------|
| 跨模块调用走接口 | 多写一个接口定义，但代码更清晰 |
| DTO 不跨模块引用 | 每个模块自己定义，稍微重复但解耦 |
| UUID 主键 | 和自增 ID 一样用，无额外成本 |
| API 路径对齐 | 只是命名规范，不影响功能 |

> **这些策略本质上是好的编码习惯，即使不拆微服务，也能让代码更健壮、更易维护。**

## 六、审计反模式 — 这些代码绝对不要写

> 以下全部来自本项目 2026-05-19 商用审计的真实发现。每一个都是已经踩过的坑。

### 6.1 竞态条件：计数器/库存用"读-改-写"

```java
// ❌ 致命：并发下必然数据错乱
Product product = productMapper.selectById(productId);
int newQty = product.getStockQty() + delta;   // 读到旧值
product.setStockQty(newQty);                   // 设置新值
productMapper.updateById(product);             // 后提交的覆盖先提交的

// ✅ 商用：一条原子 SQL 解决
int rows = jdbcTemplate.update(
    "UPDATE product SET stock_qty = stock_qty + ? WHERE id = ? AND stock_qty + ? >= 0",
    delta, productId, delta);
if (rows == 0) throw new BusinessException(400, "库存不足");
```

**同样适用于：** 优惠券发放计数、余额扣减、积分增减、任何 `count + 1` 操作。

### 6.2 NullPointerException：不判空就用

```java
// ❌ NPE：auth 为 null 时 substring(7) 直接崩
String token = request.getHeader("Authorization").substring(7);

// ✅ 防御：先判空
String auth = request.getHeader("Authorization");
if (auth == null || !auth.startsWith("Bearer ")) {
    return Result.error(401, "未登录或Token已过期");
}
String token = auth.substring(7);
```

### 6.3 @Async 自调用：AOP 代理不生效

```java
// ❌ Spring AOP 不拦截 this.method()，@Async 不生效
@Async
public void saveLogAsync(AuditLogEntry entry) { ... }
// 同类的另一个方法调用 this.saveLogAsync(entry);  ← 同步执行！

// ✅ 异步方法必须抽到独立的 Service
@Service
public class AuditLogAsyncService {
    @Async
    public void save(AuditLogEntry entry) { ... }
}
// 调用方注入 AuditLogAsyncService 调用
```

### 6.4 前端无错误处理

```javascript
// ❌ 收款失败无提示，用户以为成功了
await consumptionApi.create(dto);
ElMessage.success("收款成功");

// ✅ 必须 try-catch
try {
  await consumptionApi.create(dto);
  ElMessage.success("收款成功");
} catch (error) {
  ElMessage.error(error.message || "收款失败，请重试");
}
```

### 6.5 金额精度丢失

```java
// ❌ intValue() 直接截断小数，168.80 → 168
order.setPointsEarned(totalAmount.intValue());

// ✅ BigDecimal 规范舍入
order.setPointsEarned(totalAmount.setScale(0, RoundingMode.HALF_UP).intValue());
```

### 6.6 循环单条 INSERT

```java
// ❌ 100 条明细 = 100 次 DB 往返
for (OrderItemDTO item : items) {
    orderItemMapper.insert(toEntity(item));
}

// ✅ 批量：1 次 DB 往返
jdbcTemplate.batchUpdate(
    "INSERT INTO consumption_order_item (order_id, item_id, item_name, item_price, quantity) VALUES (?, ?, ?, ?, ?)",
    items, items.size(),
    (ps, item) -> { ps.setLong(1, orderId); ps.setLong(2, item.getItemId()); /*...*/ }
);
```

### 6.7 Excel 导入无校验

```java
// ❌ 无文件大小/类型/行级校验，10GB 文件打爆内存
@PostMapping("/import")
public Result<?> importExcel(@RequestParam("file") MultipartFile file) {

if (!file.getOriginalFilename().endsWith(".xlsx")) return Result.error(400, "仅支持.xlsx");
try (Workbook wb = WorkbookFactory.create(file.getInputStream())) { /*...*/ }
```
