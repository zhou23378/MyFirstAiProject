# PRD 目录

这里存放单个大功能的产品需求文档。`SPEC.md` 管稳定系统契约，`docs/21` 管路线图，`TODO.md` 管当前执行清单，PRD 只回答某个功能为什么做、做成什么样、怎么验收。

## 什么时候需要 PRD

需要 PRD：

- 新增一个完整业务模块。
- 改造高频核心流程，例如预约日历、真实支付、营销活动、AI 流失预测。
- 涉及用户体验、业务目标、范围取舍和验收标准的功能。

不需要 PRD：

- `.gitignore`、CI、数据脱敏等工程修复。
- 小 bug 修复。
- 已有功能的窄范围样式调整。

## 命名

```text
PRD-预约日历看板.md
PRD-真实支付接入.md
PRD-Redis生产化.md
PRD-AI流失预测.md
```

## 模板

```md
# PRD-功能名

## 1. 问题

说明用户或业务遇到的真实问题。没有证据时明确写“当前为判断，尚无用户证据”。

## 2. 目标

- 指标 1：从 X 到 Y。
- 指标 2：从 X 到 Y。

## 3. 目标用户

具体是谁会使用或受益。

## 4. 需求

- P0：没有它不能发布。
- P1：可以发布，但体验明显降级。
- P2：锦上添花，后续再做。

## 5. 成功指标

- 主指标：
- 反指标：

## 6. 范围外

明确本轮不做什么，以及为什么。

## 7. 未决问题

- 问题：
- 负责人：
- 决策时间：
```
