## 九、生产级架构差距评审（2026-05-23）

> 评审视角：高并发、多实例部署、故障恢复、可观测性
> 当前架构：单机单体，docker-compose 单副本部署
> 目标：识别从"代码级合规"到"生产级可用"之间的架构鸿沟

### 9.1 差距总览

| 领域 | 当前状态 | 差距 | 影响面 |
|------|---------|------|--------|
| **Redis** | 完全缺失 | 缓存、分布式锁、限流、会话存储 | 全部模块 |
| **消息队列** | 完全缺失 | 异步解耦、削峰填谷、事件驱动 | 订单/支付/通知 |
| **缓存层** | 无 | 每次查询命中DB，无 `@Cacheable` | 会员/服务/商品查询 |
| **分布式锁** | MySQL `GET_LOCK` | 会话绑定、无看门狗、无自动续期 | 6个定时任务 |
| **限流** | 内存 `ConcurrentHashMap` | 重启丢失、多实例不共享 | 登录/短信接口 |
| **线程池** | `SimpleAsyncTaskExecutor` | 每个 `@Async` 创建新线程，无上限 | 审计日志 |
| **数据库** | 单节点 HikariCP 默认 | 无读写分离、无连接池调优 | 全部查询 |
| **会话管理** | 客户端每次查DB | 无 TTL 自动过期、每请求1次查询 | H5客户端 |
| **优雅关闭** | 无 | 无 `@PreDestroy`、无 draining | 部署/滚动更新 |
| **可观测性** | 仅 TraceId + Actuator | 无指标、无分布式追踪、无告警 | 运维排查 |

### 9.2 详细分析

#### 9.2.1 Redis 缺失 — P0 生产上线前必须

**当前做法 vs 生产要求：**

| 场景 | 当前 | 问题 | 应改为 |
|------|------|------|--------|
| 分布式锁（6个Scheduler） | `jdbcTemplate.update("SELECT GET_LOCK(?,0)")` | 连接断开锁自动释放；无续期机制；加重DB负担 | `Redisson.getLock().tryLock()` 带 WatchDog 自动续期 |
| 限流计数器 | `ConcurrentHashMap<String, WindowCounter>` + `synchronized` | 多实例各自计数，实际限制=单实例×N；重启清零 | Redis `INCR` + `EXPIRE` 或 Redisson `RRateLimiter` |
| 支付回调幂等 | `ConcurrentHashMap<String, Boolean>.putIfAbsent()` | 同上，多实例可重复处理同一回调 | `SETNX callback:{transactionId} EX 86400` |
| 验证码存储 | `customer_session` 表 `code_hash` 列 | 每次验证都需要 DB 查询 | `SETEX sms:{phone} 300 {code}` + TTL 自动过期 |
| 客户会话 | `customer_session` 表，每次请求查DB | 高频查询，无 TTL 自动失效 | `SETEX session:{token} 7200 {memberId}` + Redis 读取 |

**涉及组件：** `spring-boot-starter-data-redis` + `Redisson`（分布式锁）或 `Lettuce`（缓存）

---

#### 9.2.2 消息队列缺失 — P1 高并发场景

**需要 MQ 的场景：**

| 场景 | 当前方案 | 高并发问题 | 建议 |
|------|---------|-----------|------|
| 订单创建后发券/积分 | 同步 `@Transactional` 内 | 事务时间长，耦合严重 | MQ 异步：订单成功→`OrderCreatedEvent`→发券+积分消费者 |
| 支付回调处理 | 同步 Controller | 回调超时会导致支付方重试 | MQ 异步确认：收到回调→发MQ→立即返回 `SUCCESS` |
| 日结计算 | 手动触发 | 月末集中日结时大量计算阻塞API | MQ 异步：日结请求→发MQ→后台逐条处理 |
| 通知推送 | 无（短信同步调用） | 短信超时阻塞登录流程 | MQ 异步：`SendSmsEvent` → 消费者调用短信SDK |
| 审计日志 | `@Async` + `SimpleAsyncTaskExecutor` | 无限创建线程，OOM风险 | MQ 异步：`AuditLogEvent`→消费者批量写库 |

**建议方案：** 初期用 `RabbitMQ`（Spring AMQP 集成成熟），后期可切换 `RocketMQ`（阿里系，事务消息支持好）

---

#### 9.2.3 缓存缺失 — P1 性能优化

**高频读、低频写的候选缓存：**

| 数据 | 特点 | 缓存策略 |
|------|------|---------|
| 服务项目列表 | 几乎不变，每次查询都走DB | `@Cacheable("services")` TTL 30min |
| 会员等级配置 | 几乎不变 | `@Cacheable("memberLevels")` TTL 1h |
| 商品分类 | 几乎不变 | `@Cacheable("productCategories")` |

**改造模式（以 CouponExpiryScheduler 为例）：**

```java
// 改造前（当前代码，第27-40行）：
String lockKey = "coupon_expiry";
Integer acquired = jdbcTemplate.queryForObject(
    "SELECT GET_LOCK(?, 0)", Integer.class, lockKey);
if (acquired == null || acquired != 1) {
    log.info("锁被其他实例持有，跳过");
    return;
}
try {
    // ... 业务逻辑 ...
} finally {
    jdbcTemplate.execute("SELECT RELEASE_LOCK('" + lockKey + "')");
}

// 改造后：
@RequiredArgsConstructor
public class CouponExpiryScheduler {
    private final RedissonClient redissonClient;
    
    @Scheduled(cron = "0 0 4 * * *")
    public void expireCoupons() {
        RLock lock = redissonClient.getLock("coupon_expiry");
        if (!lock.tryLock()) {
            log.info("锁被其他实例持有，跳过");
            return;
        }
        try {
            // ... 业务逻辑不变 ...
        } finally {
            lock.unlock();
        }
    }
}
```

#### 9B. 限流改造：内存 ConcurrentHashMap → Redis 滑动窗口

**涉及文件：** `security/RateLimitFilter.java`（`ConcurrentHashMap` 第42行，`WindowCounter` 第58-82行，Cleanup Scheduler 第45-52行）

```java
// 改造前：ConcurrentHashMap<String, WindowCounter>
// 改造后：Redis Lua 脚本实现滑动窗口

@Component
@RequiredArgsConstructor
public class RateLimitFilter extends OncePerRequestFilter {
    
    private final StringRedisTemplate redisTemplate;
    
    // Lua 脚本：滑动窗口计数 + 自动过期
    private static final String SLIDING_WINDOW_LUA = """
        local key = KEYS[1]
        local window = tonumber(ARGV[1])
        local limit = tonumber(ARGV[2])
        local now = redis.call('TIME')[1]
        local min = now - window
        redis.call('ZREMRANGEBYSCORE', key, 0, min)
        local count = redis.call('ZCARD', key)
        if count >= limit then
            return 0
        end
        redis.call('ZADD', key, now, now .. '_' .. count)
        redis.call('EXPIRE', key, window + 1)
        return 1
        """;
    
    private final DefaultRedisScript<Long> script 
        = new DefaultRedisScript<>(SLIDING_WINDOW_LUA, Long.class);
    
    @Override
    protected void doFilterInternal(...) {
        // ...
        String key = "rate:" + path + ":" + clientIp;
        Long allowed = redisTemplate.execute(script, List.of(key), 
            String.valueOf(windowSeconds), String.valueOf(limit));
        if (allowed == 0) {
            response.setStatus(429);
            return;
        }
        chain.doFilter(request, response);
    }
    // 删除：WindowCounter 内部类、Cleanup Scheduler
}
```

### Phase 13 — 生产部署（多实例 + 负载均衡）

**文件：** `docker-compose.yml` — 多副本 + Nginx 上游

```yaml
services:
  backend:
    build: ./backend
    deploy:
      replicas: 3
    environment:
      REDIS_HOST: redis
      RABBITMQ_HOST: rabbitmq
    # ... 移除端口暴露，仅通过 Nginx 访问

  nginx:
    image: nginx:alpine
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf
    ports:
      - "80:80"
    depends_on:
      - backend

  redis:
    image: redis:7-alpine
    volumes:
      - redis_data:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      retries: 3

  rabbitmq:
    image: rabbitmq:3-management-alpine
    environment:
      RABBITMQ_DEFAULT_USER: ${RABBITMQ_USER}
      RABBITMQ_DEFAULT_PASS: ${RABBITMQ_PASS}
    volumes:
      - rabbitmq_data:/var/lib/rabbitmq
    healthcheck:
      test: ["CMD", "rabbitmq-diagnostics", "check_port_connectivity"]

volumes:
  redis_data:
  rabbitmq_data:
```

**文件：** `nginx.conf` — 增加 upstream 负载均衡

```nginx
upstream backend_cluster {
    least_conn;
    server backend:8080 max_fails=3 fail_timeout=30s;
    # replicas: 3 → docker-compose 自动创建 backend_1, backend_2, backend_3
}

server {
    location /api/ {
        proxy_pass http://backend_cluster;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_read_timeout 30s;
    }
}
```
