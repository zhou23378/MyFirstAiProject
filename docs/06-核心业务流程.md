# 核心业务流程

> 本文档记录关键业务逻辑的完整执行步骤。修改支付、积分、等级相关代码前必须阅读本文档。

> 本文档记录 Phase 7+ 新增模块的核心业务逻辑。基础流程（消费、退款、积分、JWT、前端数据流）见 [docs/04-API接口详细文档.md](docs/04-API接口详细文档.md) 和 [docs/07-前端路由与组件清单.md](docs/07-前端路由与组件清单.md)。

## 一、修改业务流程的安全须知

| 场景 | 必须做的事 |
|------|-----------|
| 修改金额计算 | 确保 `RoundingMode.HALF_UP`，防止浮点精度问题 |
| 修改积分规则 | 同步检查 `addPointsAndUpgrade` 中的升级阈值匹配 |
| 修改余额扣减 | 确保在事务内，防止超扣 |
| 新增支付方式 | 同步更新 `PayMethodEnum` + 前端支付方式列表 |
| 修改等级折扣 | 确保 `discountRate` 范围 0.00~1.00，防止负价格 |
| 修改 Token 逻辑 | 同步更新拦截器白名单 + 前端 401 处理 |

## 二、技师状态变更流程（Phase 7）

**入口：** `POST /api/technician-status/change` → `TechnicianStatusServiceImpl.changeStatus()`

**代码位置：** `technician/service/impl/TechnicianStatusServiceImpl.java`

### 执行步骤

```
第 1 步：校验员工存在
  ├── employeeMapper.selectById(dto.employeeId)
  ├── 不存在 → throw BusinessException("员工不存在")
  └── 存在 → 继续

第 2 步：查询或创建技师状态
  ├── technicianStatusMapper.selectOne(employeeId)
  ├── 不存在 → 自动创建（status=OFF_DUTY）
  └── 存在 → 继续

第 3 步：校验状态变更合法性
  ├── 当前 OFFDUTY → 只能变更为 AVAILABLE
  ├── 当前 AVAILABLE → 可变更为 BUSY / BREAK / OFF_DUTY
  ├── 当前 BUSY → 可变更为 AVAILABLE（服务完成）/ BREAK
  ├── 当前 BREAK → 可变更为 AVAILABLE
  └── 不合法 → throw BusinessException

第 4 步：更新状态
  ├── status = dto.status
  ├── lastStatusChanged = LocalDateTime.now()
  ├── 可选：currentCustomerName / currentServiceName
  └── updateById(technicianStatus)
```

### 状态机图

```
OFF_DUTY ──→ AVAILABLE ──→ BUSY ──→ AVAILABLE
                │   ↑         │
                │   └─────────┘
                ↓
              BREAK ──→ AVAILABLE
```

## 三、轮牌排队 + 分配技师流程（Phase 7）

**入队入口：** `POST /api/service-queue/enqueue` → `ServiceQueueServiceImpl.enqueue()`

**分配入口：** `POST /api/service-queue/assign` → `ServiceQueueServiceImpl.assign()`

**代码位置：** `queue/service/impl/ServiceQueueServiceImpl.java`

### 入队步骤

```
第 1 步：校验会员存在
  └── memberMapper.selectById(dto.memberId)

第 2 步：计算排队号
  ├── SELECT MAX(queue_number) FROM service_queue WHERE DATE(create_time) = CURDATE()
  ├── 没有 → queueNumber = 1
  └── 有 → queueNumber = max + 1

第 3 步：插入排队记录
  ├── status = 1（等待中）
  ├── waitSince = LocalDateTime.now()
  └── insert(serviceQueue)
```

### 分配技师步骤（@Transactional）

```
第 1 步：校验排队记录
  ├── serviceQueueMapper.selectById(queueId)
  ├── status != 1 → throw BusinessException("该排队已处理")
  └── status == 1 → 继续

第 2 步：校验技师可用
  ├── technicianStatusMapper.selectOne(employeeId)
  ├── status == null || status != "AVAILABLE"
  │   └── throw BusinessException("技师当前不可用")
  └── status == AVAILABLE → 继续

第 3 步：更新排队记录
  ├── queue.status = 2（已分配）
  ├── queue.assignedEmployeeId = employeeId
  └── updateById(queue)

第 4 步：更新技师状态
  ├── ts.status = "BUSY"
  ├── ts.currentCustomerName = queue.memberName
  ├── ts.currentServiceName = queue.serviceItemName
  ├── ts.lastStatusChanged = now
  └── technicianStatusMapper.updateById(ts)

第 5 步：创建服务计时器
  ├── ServiceTimer 插入一条
  │   ├── queueId = queue.id
  │   ├── employeeId = employeeId
  │   ├── memberId = queue.memberId
  │   ├── serviceItemName = queue.serviceItemName
  │   ├── plannedDuration = 30（默认，后续可改）
  │   ├── status = 1（进行中）
  │   └── startedAt = now
  └── serviceTimerMapper.insert(timer)
```

**事务边界：** 整个 assign() 标记 @Transactional，5 步中任意一步失败则全部回滚。保证排队状态、技师状态、计时器三者一致性。

## 四、服务计时与提醒流程（Phase 7）

**入口：** `ServiceTimerController` → `ServiceTimerServiceImpl`

**代码位置：** `timer/service/impl/ServiceTimerServiceImpl.java`

### 计时控制

```
POST /api/service-timer/start    → status=1, startedAt=now
POST /api/service-timer/{id}/pause   → status=2, pausedAt=now
POST /api/service-timer/{id}/resume  → 修正 startedAt 补偿暂停时长, status=1
POST /api/service-timer/{id}/complete → 计算 actualDuration, status=3, 恢复技师 AVAILABLE
```

### 暂停恢复时间修正逻辑

```
resume():
  pauseDuration = Duration.between(pausedAt, now).toMinutes()
  timer.startedAt = timer.startedAt.plusMinutes(pauseDuration)
  timer.status = 1
  timer.pausedAt = null
```

### 前端提醒逻辑（ServiceTimer.vue）

```
每 1 秒计算:
  elapsed = now - startedAt (扣除暂停时间)
  progress = elapsed / plannedDuration

  if progress >= 1.0 && !alert100Sent:
    → emit('alert-100') → 红色脉冲动画 + 浏览器 Notification("服务已超时")
  else if progress >= 0.8 && !alert80Sent:
    → emit('alert-80') → 黄色脉冲动画 + 浏览器 Notification("服务即将完成")
```

## 五、财务日结对账流程（Phase 7）

**入口：** `DailyCloseController` → `DailyCloseServiceImpl`

**代码位置：** `dailyclose/service/impl/DailyCloseServiceImpl.java`

### 系统汇总逻辑（getTodaySummary）

```
JdbcTemplate 执行聚合 SQL:
  SELECT
    COALESCE(SUM(CASE WHEN pay_method = 1 THEN pay_amount ELSE 0 END), 0) AS cash,
    COALESCE(SUM(CASE WHEN pay_method = 3 THEN pay_amount ELSE 0 END), 0) AS wechat,
    COALESCE(SUM(CASE WHEN pay_method = 4 THEN pay_amount ELSE 0 END), 0) AS alipay,
    COALESCE(SUM(balance_used), 0) AS balance,
    COALESCE(SUM(CASE WHEN pay_method = 6 THEN pay_amount ELSE 0 END), 0) AS card
  FROM consumption_order
  WHERE status = 1 AND DATE(create_time) = ?

systemTotal = cash + wechat + alipay + balance + card
```

### 日结提交流程（submitEntry）

```
第 1 步：查询当天是否已有日结
  ├── 有草稿 → 更新人工录入字段
  └── 无 → 插入新日结记录

第 2 步：计算差异
  └── diffAmount = systemTotal - manualTotal

第 3 步：锁定（POST /api/daily-close/{id}/lock）
  ├── status = 1（已锁定）
  ├── lockedBy = 当前用户
  ├── lockedAt = now
  └── 锁定后不可再修改
```

## 六、RFM 分层计算流程（Phase 7）

**入口：** `RfmScheduler.rfmCalculate()` — 每周一凌晨 2:00

**代码位置：** `marketing/scheduler/RfmScheduler.java`

### 计算步骤

```
第 1 步：加载全量会员
  └── memberMapper.selectList(null)

第 2 步：计算每个会员的 R/F/M
  ├── R = 距今天数（从 last_visit_date 算，无记录=999）
  ├── F = 近 90 天消费次数（COUNT consumption_order WHERE member_id AND create_time > 90天前）
  └── M = 累计消费金额（total_spent）

第 3 步：计算中位数
  ├── R 中位数（所有会员 R 排序取中）
  ├── F 中位数
  └── M 中位数

第 4 步：8 段分层
  ├── R 低 F 高 M 高 → CHAMPIONS（冠军）
  ├── R 低 F 高 M 低 → LOYAL（忠诚）
  ├── R 低 F 低 M 高 → POTENTIAL（潜力）
  ├── R 低 F 低 M 低 → NEW（新客）
  ├── R 高 F 高 M 高 → AT_RISK（流失风险）
  ├── R 高 F 高 M 低 → NEED_ATTENTION（需关注）
  ├── R 高 F 低 M 高 → DORMANT_HIGH（沉睡高价值）
  └── R 高 F 低 M 低 → LOST（已流失）

第 5 步：批量更新
  └── memberMapper.updateById(member) 逐条更新 rfm_segment
```

## 九、预约提醒自动化流程（Phase C）

**入口：** `AppointmentReminderScheduler.processAppointmentReminders()` — 每天上午 9:00

**代码位置：** `notification/scheduler/AppointmentReminderScheduler.java`

### 执行步骤

```
第 1 步：获取分布式锁（防多实例重复执行）
  ├── SELECT GET_LOCK('appointment_reminder_scheduler', 0)
  ├── 获取失败 → log.info("lock not acquired, skip") → 结束
  └── 获取成功 → 进入第 2 步

第 2 步：查询当天预约（status=1）
  └── SELECT a.*, m.name, m.phone, si.name AS service_name
      FROM appointment a
      JOIN member m ON a.member_id = m.id
      JOIN service_item si ON a.service_item_id = si.id
      WHERE a.appointment_date = ? AND a.status = 1

第 3 步：逐条发送提醒短信（try-catch 逐条容错）
  ├── 跳过无手机号的预约
  ├── 组装短信内容："亲爱的{name}，您预约的{date} {time} {service_name}服务即将开始，请准时到店。"
  ├── smsService.send(phone, APPOINTMENT_REMINDER, templateCode, content, memberId)
  └── 失败 → log.error(..., e) 继续下一条

第 4 步：释放分布式锁
  └── SELECT RELEASE_LOCK('appointment_reminder_scheduler')
```

## 十一、挂单/取单流程（B9, 2026-05-23）

**入口：** `POST /api/consumption/suspend` → `PUT /api/consumption/{id}/resume` → `ConsumptionOrderServiceImpl`

**代码位置：** `consumption/service/impl/ConsumptionOrderServiceImpl.java`

### 挂单（suspendOrder）

```
第 1 步：校验会员存在
  └── memberMapper.selectById(dto.memberId) → 不存在则抛异常

第 2 步：获取会员等级折扣率
  └── memberLevelMapper.selectById(member.level)

第 3 步：计算原总金额 + 应用等级折扣
  └── BigDecimal.setScale(2, HALF_UP)

第 4 步：创建挂单（status=0）
  ├── baseMapper.insert(order) — status=0（SUSPENDED）
  ├── 计算提成（commission_amount）
  └── jdbcTemplate.batchUpdate — 插入明细

挂单特点：
  - 不扣余额/积分
  - 不处理支付
  - 不核销优惠券
  - 可被取单恢复并完成支付
```

### 取单（resumeOrder）

```
第 1 步：加载挂单
  ├── baseMapper.selectById(id)
  ├── status≠0 → 抛异常（ORDER_ALREADY_PAID / BAD_REQUEST）
  └── status=0 → 继续

第 2 步：更新支付信息
  ├── 组合支付：paymentDetailService.batchInsert()
  └── 优惠券核销：原子 UPDATE coupon SET status = 2 WHERE id = ? AND status = 1

第 3 步：原子更新会员（A1 + H1）
  ├── 有余额扣减：UPDATE member SET balance = balance - ?, points = points + ?, ...
  │   WHERE id = ? AND balance >= ?
  └── 无余额扣减：UPDATE member SET points = points + ?, ...

第 4 步：等级自动升级
  └── memberLevelMapper.selectList → 匹配最高等级 → jdbcTemplate.update level

第 5 步：原子状态转换（A2）
  └── UPDATE consumption_order SET status = 1, ... WHERE id = ? AND status = 0
```

## 十二、采购订单审批+入库流程（G7, 2026-05-23）

**入口：** `PurchaseOrderController` → `PurchaseOrderServiceImpl`

**状态流转：** 0(草稿) → 1(已提交) → 2(已审核) → 3(已收货) | 0/1 → 4(已取消)

### 收货（receive）

```
第 1 步：原子状态转换（A2）
  └── UPDATE purchase_order SET status = 3, received_time = ? WHERE id = ? AND status = 2

第 2 步：逐条入库（for each item）
  ├── 原子增加库存：UPDATE product SET stock_qty = stock_qty + ? WHERE id = ?
  ├── 写库存流水：INSERT INTO stock_record (type=1, 入库)
  ├── 更新已收数量：UPDATE purchase_order_item SET received_qty = ?
  └── 每步检查 affected rows == 0（A3）

入库特点：
  - 库存原子增加（A1），无需下限守卫（入库不会超限）
  - stock_record.type = 1（入库）
```

## 十三、退货审批+出库流程（G8, 2026-05-23）

**入口：** `ReturnOrderController` → `ReturnOrderServiceImpl`

**状态流转：** 0(草稿) → 1(已提交) → 2(已审核) → 3(已完成) | 1→4(驳回) | 0→4(取消)

### 完成退货（complete）

```
第 1 步：原子状态转换（A2）
  └── UPDATE return_order SET status = 3, completed_time = ? WHERE id = ? AND status = 2

第 2 步：逐条出库（for each item）
  ├── 原子扣减库存（H1 下限守卫）：
  │   UPDATE product SET stock_qty = stock_qty - ? WHERE id = ? AND stock_qty >= ?
  ├── 写库存流水：INSERT INTO stock_record (type=3, 退货)
  └── 每步检查 affected rows == 0（A3）

退货特点：
  - 库存原子扣减，带 H1 下限守卫（防止超额退货）
  - stock_record.type = 3（退货出库）
```

## 十四、提成结算流程（F5, 2026-05-23）

**入口：** `CommissionSettlementController` → `CommissionSettlementServiceImpl`

**状态流转：** 0(草稿) → 1(已确认) → 2(已支付)

### 生成结算（create）

```
第 1 步：校验周期
  └── periodStart ≤ periodEnd，否则 BAD_REQUEST

第 2 步：查重
  └── SELECT COUNT(*) FROM commission_settlement WHERE period_start = ? AND period_end = ?
  └── count > 0 → COMMISSION_SETTLEMENT_PERIOD_EXISTS

第 3 步：聚合消费订单
  └── SELECT o.employee_id, e.name, COUNT(DISTINCT o.id), SUM(o.commission_amount)
      FROM consumption_order o JOIN employee e
      WHERE o.employee_id IS NOT NULL AND o.status = 1
      AND o.create_time >= ? AND o.create_time < ?
      GROUP BY o.employee_id, e.name

第 4 步：批量插入结算单
  └── jdbcTemplate.batchUpdate(List<Object[]> batchArgs)
  └── 逐条检查 successCount（A3）

第 5 步：jdbcTemplate 回读（A4）
  └── jdbcTemplate.queryForList() 替代 baseMapper.selectList()
```

### 确认/支付

```
confirm: UPDATE commission_settlement SET status = 1, confirmed_at = NOW() WHERE id = ? AND status = 0
pay:     UPDATE commission_settlement SET status = 2, paid_at = NOW() WHERE id = ? AND status = 1
delete:  DELETE FROM commission_settlement WHERE id = ? AND status = 0

所有操作必须有旧状态守卫（A2）+ affected rows 检查（A3）
```

## 十五、积分兑换流程（D10, 2026-05-23）

**入口（H5）：** `POST /api/customer/portal/points/exchange` → `PointsExchangeRecordServiceImpl.exchange()`

**代码位置：** `points/service/impl/PointsExchangeRecordServiceImpl.java`

### 执行步骤（@Transactional）

```
第 1 步：查询会员
  └── jdbcTemplate.queryForMap("SELECT id, name, phone, points FROM member WHERE id = ?")
  └── 不存在 → MEMBER_NOT_FOUND

第 2 步：查询商品
  └── jdbcTemplate.queryForMap("SELECT id, name, points_price, stock_qty, status FROM points_product WHERE id = ?")
  └── 不存在 → POINTS_PRODUCT_NOT_FOUND
  └── status ≠ 1 → POINTS_PRODUCT_OFF_SHELF

第 3 步：初步校验（快速失败，友好错误信息）
  ├── stockQty < quantity → POINTS_PRODUCT_STOCK_INSUFFICIENT
  └── currentPoints < totalPointsCost → MEMBER_POINTS_INSUFFICIENT

第 4 步：原子扣库存（A1 + H1）
  └── UPDATE points_product SET stock_qty = stock_qty - ?, exchanged_count = exchanged_count + ?
      WHERE id = ? AND stock_qty >= ?
  └── rows == 0 → POINTS_PRODUCT_STOCK_INSUFFICIENT（并发冲突）

第 5 步：原子扣积分（A1 + H1）
  └── UPDATE member SET points = points - ? WHERE id = ? AND points >= ?
  └── rows == 0 → MEMBER_POINTS_INSUFFICIENT（并发冲突）

第 6 步：插入兑换记录
  └── INSERT INTO points_exchange_record (..., status=0, exchanged_at=NOW())
  └── rows == 0 → INTERNAL_ERROR

第 7 步：jdbcTemplate 回读（A4）
  └── jdbcTemplate.queryForList() 替代 baseMapper.selectList()

并发安全设计：
  - 第 3 步的初步校验提供友好错误信息（非必须，但提升 UX）
  - 第 4-5 步的原子 SQL 是真正的并发防线（A1 + H1）
  - 两个并发兑换最后一件商品：一个成功，另一个在第 4 步 rows==0
```

### 标记已领取（claim）

```
UPDATE points_exchange_record SET status = 1, claimed_at = NOW()

### 16.1 状态机

```
GroupBuyOrder (团单):

  1(待成团) ──团成──> 2(已成团) ──核销──> 3(已核销)

### 16.2 开团流程

**入口：** `POST /api/customer/portal/group-buy/orders` → `GroupBuyOrderServiceImpl.create(memberId, req)`

```
第 1 步：查询会员
  └── memberMapper.selectById(memberId)
  └── 不存在 → UNAUTHORIZED

第 2 步：查询模板并校验
  ├── jdbcTemplate.queryForMap("SELECT * FROM group_buy_template WHERE id = ?")
  ├── 不存在 → GROUP_BUY_TEMPLATE_NOT_FOUND
  ├── status ≠ 1 → GROUP_BUY_TEMPLATE_NOT_ACTIVE
  ├── startTime/endTime 范围校验 → GROUP_BUY_TEMPLATE_NOT_IN_TIME
  └── totalQty > 0 AND issuedQty >= totalQty → GROUP_BUY_TEMPLATE_EXHAUSTED

第 3 步：余额检查（快速失败）
  └── member.balance < template.groupPrice → GROUP_BUY_MEMBER_BALANCE_INSUFFICIENT

第 4 步：@Transactional — 原子操作块
  ├── 4a. 原子扣模板库存（A1 + H1）
  │   └── UPDATE group_buy_template SET issued_qty = issued_qty + 1
  │       WHERE id = ? AND (total_qty = 0 OR issued_qty < total_qty)
  │   └── rows == 0 → GROUP_BUY_TEMPLATE_EXHAUSTED（并发）
  │
  ├── 4b. 原子扣余额（A1 + H1 + F2）
  │   └── UPDATE member SET balance = balance - ?
  │       WHERE id = ? AND balance >= ?
  │   └── rows == 0 → GROUP_BUY_MEMBER_BALANCE_INSUFFICIENT（并发）
  │
  ├── 4c. 插入余额流水（F2）
  │   └── INSERT INTO recharge_record (member_id, amount=-groupPrice, type=拼团支付, ...)
  │
  ├── 4d. 创建团单
  │   └── INSERT INTO group_buy_order (order_no=GB-xxx, status=1, current_size=1,
  │       expire_time=NOW()+expireHours, ...)
  │
  ├── 4e. 团长参团
  │   └── INSERT INTO group_buy_participant (is_leader=1, status=1, join_price=groupPrice, ...)
  │
  └── 4f. jdbcTemplate 回读团单（A4，避免 MyBatis 缓存旧数据）
      └── getDetail(id) → 返回 GroupBuyOrderVO

第 5 步：返回 GroupBuyOrderVO（含参团列表）
```

### 16.3 参团流程

**入口：** `POST /api/customer/portal/group-buy/orders/{id}/join` → `GroupBuyOrderServiceImpl.join(orderId, memberId)`

```
第 1 步：查询会员
  └── memberMapper.selectById(memberId)

第 2 步：查询团单并校验
  ├── jdbcTemplate.queryForMap("SELECT * FROM group_buy_order WHERE id = ?")
  ├── 不存在 → GROUP_BUY_ORDER_NOT_FOUND
  ├── status ≠ 1 → GROUP_BUY_ORDER_CLOSED
  ├── currentSize >= groupSize → GROUP_BUY_ORDER_FULL
  └── expireTime < NOW() → GROUP_BUY_ORDER_EXPIRED

第 3 步：重复参团检查
  └── SELECT COUNT(*) FROM group_buy_participant WHERE order_id = ? AND member_id = ?
  └── count > 0 → GROUP_BUY_ALREADY_JOINED

第 4 步：余额检查
  └── member.balance < order.groupPrice → GROUP_BUY_MEMBER_BALANCE_INSUFFICIENT

第 5 步：@Transactional — 原子操作块
  ├── 5a. 原子扣余额（A1 + H1 + F2）
  │   └── UPDATE member SET balance = balance - ? WHERE id = ? AND balance >= ?
  │
  ├── 5b. 插入余额流水（F2）
  │
  ├── 5c. 原子增人数 + 条件成团（A1 + A2）★ 核心并发点
  │   └── UPDATE group_buy_order
  │       SET current_size = current_size + 1,
  │           status = IF(current_size + 1 >= group_size, 2, status),
  │           complete_time = IF(current_size + 1 >= group_size, NOW(), NULL)
  │       WHERE id = ? AND current_size < group_size AND status = 1
  │   └── rows == 0 → GROUP_BUY_ORDER_FULL 或 GROUP_BUY_ORDER_CLOSED（并发）
  │
  ├── 5d. 插入参团记录
  │   └── INSERT INTO group_buy_participant
  │       (status = 当前人数 >= groupSize ? 2 : 1, is_leader=0, ...)
  │
  ├── 5e. 若成团：批量更新参与者状态（A2）
  │   └── UPDATE group_buy_participant
  │       SET status = 2, complete_time = NOW()
  │       WHERE order_id = ? AND status = 1
  │
  └── 5f. jdbcTemplate 回读（A4）
      └── getDetail(orderId) → 返回更新后的 GroupBuyOrderVO

第 6 步：返回 GroupBuyOrderVO
```

**并发安全设计：**
- 第 5c 步的 `WHERE current_size < group_size AND status = 1` 是真正的并发防线
- 两个顾客同时抢最后一个名额：一个成功（rows=1），另一个失败（rows=0 → 团已满员）
- IF() 函数在 MySQL 中原子执行，无需应用层判断

第 1 步：原子状态转换（A2）

### 16.5 过期自动退款流程

**入口：** `GroupBuyExpiryScheduler.expireOrders()` — `@Scheduled(cron = "0 0 * * * *")` 每小时

```
第 0 步：获取多实例锁
  ├── SELECT GET_LOCK('group_buy_expiry', 0)
  ├── 获取失败 → log.info("lock not acquired, skip") → 结束
  └── 获取成功 → 进入第 1 步

第 1 步：过期团单（E1 幂等 SQL）
  └── UPDATE group_buy_order SET status = 4
      WHERE status = 1 AND expire_time < NOW()
  └── SQL 的 WHERE status=1 自身保证幂等

第 2 步：标记退款参团记录（E1 幂等 SQL）
  └── UPDATE group_buy_participant SET status = 4
      WHERE status = 1
      AND order_id IN (SELECT id FROM group_buy_order WHERE status = 4)

第 3 步：逐条退余额（E2 逐条容错）
  ├── 查询：SELECT * FROM group_buy_participant
  │   WHERE status = 4 AND refund_time IS NULL
  │
  └── 遍历每条记录：
      try {
          INSERT INTO recharge_record (type=拼团退款, amount=+joinPrice, remark="拼团退款-超时未成团")
          UPDATE member SET balance = balance + joinPrice WHERE id = ?
          UPDATE group_buy_participant SET refund_time = NOW()
              WHERE id = ? AND refund_time IS NULL
      } catch (Exception e) {
          log.error("退款失败: participantId={}", id, e)
          // 单条失败继续处理下一条
      }

第 4 步：释放多实例锁
  └── 事务完成后执行 SELECT RELEASE_LOCK('group_buy_expiry')

并发安全设计：
  - 第 0/4 步使用 MySQL 命名锁，防止多实例同时处理同一批退款
  - 锁释放注册在事务完成回调中，避免锁先释放、事务尚未提交的并发窗口
  - 第 1-2 步的 SQL 带状态守卫，重复执行无副作用（E1）
  - 第 3 步的退款 UPDATE member SET balance = balance + ? 是原子加法（A1）
  - `refund_time IS NULL` 是单条退款幂等标记，最终标记 UPDATE 必须检查 rows
  - 逐条 try-catch 确保一条失败不影响其他退款（E2）
```
