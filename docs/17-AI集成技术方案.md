## 一、AI 集成架构总览

### 1.1 架构全景图


### 1.2 核心原则

**Java 做业务，Python 做 AI，HTTP 做桥梁。**

| 角色    | 技术                 | 职责                                   |
| ------- | -------------------- | -------------------------------------- |
| 前端    | Vue 3 + Element Plus | 渲染 AI 结果，发送自然语言查询         |
| BFF 层  | Spring Boot 3.2      | 鉴权、参数校验、编排业务逻辑 + AI 结果 |
| AI 服务 | Python FastAPI       | 模型加载、向量检索、预测推理、LLM 调用 |

### 1.3 为什么是这个方案——工程决策记录

| 候选方案                               | 判定     | 理由                                                                                               |
| -------------------------------------- | -------- | -------------------------------------------------------------------------------------------------- |
| **Java ProcessBuilder 调 Python 脚本** | ❌ 否决  | 每次请求启动进程，模型加载 2-3 秒延迟；并发时内存爆炸；stdout 解析脆弱                             |
| **Spring Boot 纯 Java**                | ❌ 否决  | BGE 模型无官方 Java 支持；需 PyTorch→ONNX 格式转换；ChromaDB 无 Java SDK；LLM 生态 Java 版严重滞后 |
| **Python FastAPI 微服务** ✅           | **采用** | Python AI 生态完整；模型常驻内存毫秒响应；后期 LLM/Agent 无缝接入；独立扩展独立调试                |

> **这不是妥协方案——2026 年业界标准做法就是 Java 系统旁挂 Python AI 服务。** 连字节跳动、美团内部都是这个架构。

### 1.4 关键设计决策

| 决策点      | 选择              | 理由                                                                                   |
| ----------- | ----------------- | -------------------------------------------------------------------------------------- |
| AI 服务框架 | FastAPI           | 异步支持、自动生成 OpenAPI 文档、比 Flask 快 3 倍                                      |
| 服务间通信  | HTTP REST         | 最简单、可观测（curl 直测）、Spring `RestClient` 原生支持 3 行代码                     |
| 鉴权        | 短期不需要        | Python 服务仅监听 `127.0.0.1:8000`，外部无法访问；鉴权由 Spring Security 在 BFF 层完成 |
| 服务发现    | 不需要            | 单机部署，IP/端口在 `application.yml` 写死                                             |
| 模型存储    | `ai/models/` 目录 | 与代码分离，`.gitignore` 忽略                                                          |
| 向量库      | ChromaDB          | 纯 Python (pip install) 零运维，Windows 友好                                           |

## 二、开发规范（每次涉及 AI 开发必读）

### 2.1 启动流程

```bash
# 终端1：Python AI 服务（必须先启）
cd ai
pip install -r ../requirements_ai.txt
uvicorn server:app --host 127.0.0.1 --port 8000 --reload

# 终端2：Spring Boot（不变）
cd backend
mvn spring-boot:run

# 终端3：前端（不变）
cd frontend
npm run dev
```

模型只在 Python 服务启动时加载一次（约 2 秒），后续所有请求毫秒级响应。

### 2.2 新增一个 AI 能力——Step by Step

当你需要新增一个 AI 能力（例如"会员流失预测"）时，按照以下步骤：

```
STEP 1: 在 Python 端新建路由文件
  ai/predict_churn.py     → 只做模型推理，不关心业务

STEP 2: 定义 API 契约（请求/响应 JSON）
  {
    "request": { "member_id": 1 },
    "response": { "member_id": 1, "churn_probability": 0.73, "risk_level": "high" }
  }

STEP 3: 在 FastAPI server.py 注册路由
  from routers import predict_churn
  app.include_router(predict_churn.router, prefix="/api/ai")

STEP 4: 在 Spring Boot 新增 Client 方法
  AiServiceClient.java → predictChurn(memberId)

STEP 5: 在业务 Controller 中调用 Client
  MemberController.java → 返回给前端包含流失风险信息

STEP 6: 前端展示（新标签/卡片/对话框）
```

### 2.3 代码兼容规则

| 规则                         | 说明                                                                                                |
| ---------------------------- | --------------------------------------------------------------------------------------------------- |
| **Python 端只管 AI 推理**    | 不访问数据库、不做权限校验、不处理业务逻辑。输入是"已经处理好的数据"，输出是"纯 AI 结果"            |
| **Spring Boot 端负责"胶水"** | 从 MySQL 查数据 → 拼成 AI 需要的格式 → 调用 AI 服务 → 拿到结果 → 整理成前端格式                     |
| **API 路径统一前缀**         | Python 端所有路由使用 `/api/ai/` 前缀，与 Spring Boot 的 `/api/` 区分（内部调用，不暴露给前端直连） |
| **错误处理**                 | Python 端返回 `{"error": "message"}` + HTTP 500；Spring Boot 端 catch 后转成统一的 `Result` 格式    |
| **AI 服务不可用时的降级**    | Spring Boot 端 try-catch，AI 服务挂了时返回空结果 + warn 日志，不影响主业务                         |

**🔬 本模块能学到的 AI 知识：**

| 知识点                       | 是什么                                                                  | 在哪里体现                                             |
| ---------------------------- | ----------------------------------------------------------------------- | ------------------------------------------------------ |
| **Embedding 向量化**         | 把文本变成一串数字（向量），语义相近的文本向量距离近                    | `model.encode("喜欢红酒")` → 512维浮点数数组           |
| **余弦相似度**               | 用两个向量夹角衡量相似度（不看长度只看方向），范围 [-1, 1]              | ChromaDB 内部自动计算，返回时带 `distance` 字段        |

| **向量数据库 vs 传统数据库** | MySQL LIKE 搜"红酒"找不到"喜欢威士忌"；向量库搜"喜欢喝酒的客户"能找到   | Phase 0 已验证：关键词搜索命中 3 条，语义搜索命中 7 条 |
| **BGE 模型选型**             | 中文语义排行榜 Top 5、仅 24MB（适合本地部署）、512 维（精度与速度平衡） | `BAAI/bge-small-zh-v1.5`                               |

| ------------ | ---------------------------------------------------------------------------------- |

| **功能**     | 输入会员 ID，输出流失概率（0~1）和风险等级（高/中/低）                             |

**🔬 本模块能学到的 AI 知识：**

| 知识点            | 是什么                                                         | 在哪里体现                                                                 |
| ----------------- | -------------------------------------------------------------- | -------------------------------------------------------------------------- |
| **特征工程**      | 从原始数据（消费记录）中提取模型能理解的"特征"                 | 例如：最近一次消费距今天数、近3个月消费频率、客单价环比下降率              |
| **XGBoost 原理**  | 集成学习——N 棵决策树投票，每棵树纠正前一棵的错误               | `xgb.XGBClassifier()` 训练后保存 → `model.predict_proba()` 推理            |
| **混淆矩阵**      | TP(真流失)、FP(误报)、TN(真留存)、FN(漏报)——四宫格评估模型好坏 | 训练脚本输出：`Confusion Matrix: [[TN, FP], [FN, TP]]`                     |
| **AUC-ROC**       | 模型区分正负样本的能力（0.5=瞎猜，1.0=完美），行业及格线 0.75  | `roc_auc_score(y_true, y_pred)`                                            |
| **过拟合**        | 模型把训练数据的噪声也学会了，新数据上表现差                   | 用 `train_test_split` 划分训练集/测试集，验证集得分≈训练集得分才是健康模型 |
| **pickle 持久化** | Python 对象序列化——把训练好的模型保存成文件，推理时直接加载    | `pickle.dump(model, "churn.pkl")` → `pickle.load(open("churn.pkl"))`       |

#### Step 2：FastAPI 路由 —— `GET /api/ai/search`

```python
# ai/server.py 中的搜索端点（实际实现）

from fastapi import FastAPI, Query
from pydantic import BaseModel

app = FastAPI(title="Salon AI Service", version="1.0.0")

class SearchResult(BaseModel):
    member_id: int
    name: str
    description: str
    similarity: float

class SearchResponse(BaseModel):
    query: str
    results: list[SearchResult]

@app.get("/api/ai/search")
def search(
    q: str = Query(..., description="Natural language search query"),
    top_k: int = Query(5, ge=1, le=20)
):
    results = collection.query(
        query_texts=[q],
        n_results=top_k,
        include=["documents", "distances", "metadatas"]
    )
    items = []
    for doc, dist, meta in zip(
        results["documents"][0],
        results["distances"][0],
        results["metadatas"][0]
    ):
        items.append(SearchResult(
            member_id=meta.get("member_id", 0),
            name=meta["display_name"],
            description=doc,
            similarity=round(max(0, (1 - dist) * 100), 1)
        ))
    return SearchResponse(query=q, results=items).model_dump()
```

> **注意：** 搜索端点直接定义在 `server.py` 中（非独立 `ai/search.py` 模块）。使用 GET 方法 + 查询参数 `q` 和 `top_k`（非 POST + JSON body）。模型和 ChromaDB client 在模块启动时加载一次。

#### Step 3：Spring Boot + 前端对接

```java
// backend/.../ai/controller/AiController.java （实际实现）
// 注意：AI 集成逻辑内联在 Controller 中，无独立的 AiServiceClient 类

@RestController
@RequestMapping("/api/ai")
public class AiController {

    private final RestClient restClient;

    public AiController(@Value("${ai.service.url:http://localhost:8000}") String aiServiceUrl) {
        SimpleClientHttpRequestFactory factory = new SimpleClientHttpRequestFactory();
        factory.setConnectTimeout(Duration.ofSeconds(5));   // 连接超时 5s
        factory.setReadTimeout(Duration.ofSeconds(30));      // 读取超时 30s
        this.restClient = RestClient.builder()
            .baseUrl(aiServiceUrl)
            .requestFactory(factory)
            .build();
    }

    @GetMapping("/search")
    public Result<AiSearchResponse> search(
            @RequestParam String q,
            @RequestParam(defaultValue = "5") int topK) {
        try {
            var resp = restClient.get()
                .uri(uri -> uri.path("/api/ai/search")
                    .queryParam("q", q)
                    .queryParam("top_k", topK)
                    .build())
                .retrieve()
                .body(AiSearchResponse.class);
            return Result.success(resp);
        } catch (Exception e) {
            return Result.error(503, "AI 搜索失败: " + e.getMessage());
        }
    }
}
```

## 五、流失预测接入方案（Phase 1-AI）

### 5.1 数据准备

从 MySQL 提取以下特征（每个会员一行）：

| 特征名                  | 含义                 | SQL 来源                              |
| ----------------------- | -------------------- | ------------------------------------- |
| `days_since_last_visit` | 最近一次消费距今天数 | `MAX(consumption.created_at)`         |
| `visit_freq_3m`         | 近 3 个月消费次数    | `COUNT(*) WHERE created_at > 3个月前` |
| `avg_amount_3m`         | 近 3 个月平均客单价  | `AVG(amount)`                         |
| `amount_trend`          | 客单价环比变化率     | (近3月 - 前3月) / 前3月               |
| `level_changed`         | 最近是否降级         | `member_level_log` 表                 |
| `total_consume`         | 历史总消费           | `SUM(amount)`                         |
| `member_level_code`     | 当前会员等级编码     | `member_level.level_code`             |

**标签（y）：**

- 流失 = 超过 60 天未消费 AND 最近 30 天无预约
- 未流失 = 不符合以上条件的在籍会员

## 六、服务推荐接入方案（Phase 2-AI）

### 6.1 协同过滤矩阵

```
          服务A  服务B  服务C  服务D
会员1     3      0       2     1
会员2     0      5       0     3
会员3     2      0       4     0
```

- 行 = 会员，列 = 服务，值 = 消费次数
- 算会员 1 和其他会员的余弦相似度
- 找到最相似会员，推荐"他们消费了但会员1没消费的"服务

### 6.2 推理路由

```python
# ai/recommend.py
@router.post("/recommend")
def recommend(member_id: int, top_k: int = 5):
    # 1. 加载/计算会员-服务矩阵
    # 2. 算当前会员与所有其他会员的余弦相似度
    # 3. 取 Top-N 相似会员
    # 4. 汇总他们的消费服务（加权），过滤当前会员已消费的
    # 5. 返回 Top-K 推荐服务 ID + 推荐理由（"和你相似的会员也喜欢..."）
```

**🔬 本节能学到的额外 AI 知识：**

| 知识点                   | 说明                                                                                                                                      |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **协同过滤 vs 内容推荐** | 协同过滤看"谁跟谁行为像"（不需要理解服务内容）；内容推荐看"服务和服务的描述像不像"（需要服务标签/描述）。本项目先做协同过滤（数据已就绪） |
| **相似度矩阵**           | 一个 N×N 矩阵，每个格子是两个会员的相似度——内存占用 O(N²)，会员数 > 10万 时需要用稀疏存储                                                 |
| **推荐理由**             | 推荐系统不只是列出结果，还要解释原因——"因为你跟会员X消费习惯相似（相似度85%），你可能也喜欢..."                                           |

### 7.3 推理路由

```python
    # 2. 调用 LLM API / 本地模型
    # 3. 解析 LLM 返回的 JSON → 提取 SQL
    # 4. 执行 SQL（在 salon 数据库上）
```

**🔬 本节能学到的额外 AI 知识：**

| 知识点                 | 说明                                                                                        |
| ---------------------- | ------------------------------------------------------------------------------------------- |
| **Prompt Engineering** | 给 LLM 的"咒语"质量直接决定 SQL 正确率——Schema 信息、示例（Few-shot）、输出格式约束缺一不可 |

## 附录 A：技术决策 Q&A

**Q：为什么不用 LangChain？**
A：LangChain 是"框架"，你的需求是"工具"。Phases 0-2 的语义搜索、XGBoost、协同过滤都是经典 ML，不需要 LangChain。Phase 3 的 LLM 场景，直接调 API 远比 LangChain 简单可控。LangChain 适合"需要串联多个 LLM 调用 + 工具 + Memory"的复杂 Agent 场景，你的系统短期内不需要。

**Q：为什么不用 Milvus？**
A：Milvus 需要 Docker/Linux 环境，Windows 上维护成本高。ChromaDB 纯 Python `pip install`，数据量小于 10 万条时性能差距可忽略。等会员破 10 万 + 需要分布式时再迁不迟。

**Q：为什么不用 Docker？**
A：当前阶段不需要。FastAPI 单进程足够，Spring Boot 也一样。Docker 是"部署"需求，不是"开发"需求。等你需要换到 Linux 服务器时再加 Dockerfile（15 分钟的事）。

**Q：AI 服务挂了怎么办？**
A：Spring Boot 端 catch 异常后返回空结果 + 降级逻辑。前端搜索框回退到传统 MySQL LIKE 搜索。不影响会员开卡、收银等核心业务。

## 附录 B：AI 术语速查表

| 术语                            | 一句话解释                                                      |
| ------------------------------- | --------------------------------------------------------------- |
| **Embedding（嵌入向量）**       | 把文字/图片变成一串数字，让计算机能"理解"语义                   |
| **向量维度**                    | 这串数字的长度。BGE-small 是 512 维，维度越高表达能力越强但越慢 |
| **余弦相似度**                  | 两个向量的"方向"有多接近。1.0=完全一致，-1.0=完全相反，0=无关   |
| **特征工程**                    | 把原始数据（消费记录）变成模型能用的"特征"（近3月消费频率）     |
| **过拟合**                      | 模型背下了训练数据，遇到新数据就懵了                            |
| **AUC-ROC**                     | 模型区分"好"和"坏"的能力，0.5=瞎猜，0.8=不错，0.95=很牛         |
| **协同过滤**                    | 推荐算法核心思想——和你口味相似的人喜欢的，你也可能喜欢          |
| **冷启动**                      | 新用户/新商品没有历史数据，推荐系统不知道该推啥                 |
| **RAG（检索增强生成）**         | 先检索相关文档，再让 LLM 基于文档回答——减少幻觉                 |
| **Prompt Engineering**          | 给 LLM 设计输入指令的艺术——Prompt 好坏决定输出质量              |
| **Text-to-SQL**                 | 自然语言转 SQL 查询——LLM 最成熟的企业应用场景之一               |
| **BFF（Backend for Frontend）** | 专门为前端定制的后端层——聚合多个服务的数据，一口给前端          |
| **服务降级**                    | 一个服务挂了，系统自动切到备用方案——保证核心功能可用            |
| **HNSW**                        | 一种快速向量检索算法——类似"六度分隔理论"，查询速度 O(log N)     |
| **Token**                       | LLM 处理文本的最小单位，一个中文字≈2 tokens，计费按 token 算    |
| **Function Calling**            | LLM 不直接执行操作，而是输出结构化 JSON，由程序来执行           |
| **XGBoost**                     | 树模型的"王者"——N 棵决策树投票，Kaggle 竞赛冠军常客             |
| **混淆矩阵**                    | 4 个数字评估分类模型：对的对、对的错、错的对、错的错            |
| **Pickle**                      | Python 的序列化工具——把训练好的模型存成文件，下次直接加载       |
| **ChromaDB**                    | AI 原生的向量数据库——存向量 + 搜向量，比 MySQL 快 100 倍        |
