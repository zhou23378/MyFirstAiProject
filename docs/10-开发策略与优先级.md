# 开发策略与优先级

> 基于 2026-05-17 全面审计结论制定，2026-05-23 更新：Phase 7 完成 + Phase 8 顾客端 H5 完成。本文档是决策依据，不是功能列表。

---

## 一、核心决策

**结论：安全加固 > 功能开发。先治根，再长叶。**

当前系统处于"功能原型 → 可商用产品"的转折点。功能完整度（60-75%）已可支撑小范围试点，但安全基线为零。如果在不安全的地基上继续盖楼，每加一层都会增加未来返工的代价。

### 决策逻辑

```
安全加固（2周） → 小范围试点（2周） → 根据反馈决定下一步
                                       ├── 商户要更多功能 → Phase 3 营销获客
                                       ├── 商户要稳定可靠 → 工程化 + 测试
                                       └── 商户要线上能力 → 小程序/在线预约
```

**关键原则：不与市场猜需求，用试点数据驱动。**

## 二、为什么安全必须先做

### 2.1 三个不可推迟的理由

| 理由 | 说明 |
|------|------|
| **架构级变更** | Spring Security 引入会改变认证/授权体系，所有 Controller、Interceptor、前端 Token 逻辑都受影响。功能越多，迁移成本越大 |
| **商业底线** | 明文密码 + 硬编码密钥意味着一次数据库泄露 = 所有商户数据裸奔。这是法律风险，不是技术债 |
| **试点前提** | 在真实商户环境中跑系统，安全是准入门槛。没有商户会把自己的会员数据放在一个密码明文存储的系统上 |

### 2.2 安全加固的边界

只做"上线必须"的安全项，不做"理想但非必须"的：

**必须（10项）：**
- BCrypt 密码加密
- 环境变量化敏感配置（DB密码、JWT密钥）
- Spring Security + JWT 认证过滤器
- JWT 嵌入角色 → RBAC 权限注解
- 登录速率限制
- CORS 配置
- 生产环境关闭 Swagger
- Token 存储从 localStorage 迁移到 httpOnly Cookie
- 审计日志（登录、退款、管理员操作）
- 退款/敏感操作增加角色校验

**暂缓（在试点后根据反馈决定）：**
- CSRF 保护（Spring Security 引入后自动获得一部分）
- HTTPS（需要域名 + 证书，试点阶段可用 IP 访问）
- 完整的安全事件告警系统

## 八、中间件引入策略（2026-05-23 制定）

> **核心决策：功能开发期不引入 Redis / RabbitMQ 等中间件。分三阶段推进。**

### 8.1 为什么不在开发期引入

| 理由 | 说明 |
|------|------|
| **迭代速度** | 每加一个中间件，每天开发循环多一步确认和排障。当前 `mvn run` + `npm run dev` 10秒进入状态，应保持 |
| **单实例足够** | 日活 < 100 场景下，单机 MySQL + 内存级限流/锁完全够用，过早引入是过度设计 |
| **代码优先** | 先保证代码结构正确（DTO/VO隔离、原子SQL、幂等），这些不依赖任何中间件。基础设施是放大器，不是替代品 |
| **Docker Compose 一键启动** | 需要时用 `docker-compose -f docker-compose.dev.yml up -d` 一行命令同时拉起 Redis + RabbitMQ，不需要手动安装 |

### 8.2 三阶段策略

```
当前（功能开发期）          上线前固化期             生产运维期
───────────────     │     ─────────────     │     ─────────
零成本治理 ✅         │     Redis 引入          │     水平扩展
@Cacheable 预留       │     MQ 异步化           │     多实例
ApplicationEvent 预留  │     可观测性             │     压测调优
配置外置              │     健康检查             │
                      │                        │
   改代码不改架构        改配置+少量代码           改部署拓扑
```

### 8.3 触发矩阵

| 触发条件 | 动作 | 紧急程度 |
|---------|------|---------|
| **现在** | 零成本治理：HikariCP 连接池 + @Async 线程池 + 优雅关闭 | ✅ 已完成 |
| **每次写新查询** | 顺手加 `@Cacheable`（当前 NoOpCacheManager，不生效但不报错） | 惯例 |
| **每次写新业务操作** | 发 `ApplicationEvent`（当前 `@EventListener` 同步，MQ就位后切 RabbitMQ） | 惯例 |
| **功能全部开发完成** | Docker Compose 启动中间件 + 集中执行 Phase 9-13（详见 docs/12 §十） | 硬性前提 |
| **部署 ≥ 2 实例** | Redis + MQ 必须就位（限流/锁/幂等/会话无法多实例共享） | 硬性前提 |
| **日活 > 100** | 优先 Redis（会话 TTL + 限流 + 缓存） | 可紧急插入 |
| **日活 > 1000** | 优先 MQ（订单异步 + 支付回调削峰 + 短信解耦） | 可紧急插入 |
| **上线商用** | Phase 9-13 全部完成 | 硬性前提 |

### 8.4 功能开发期的代码预留模式

**模式 1：缓存预留 — `@Cacheable` + NoOpCacheManager**

```java
// 当前配置（Redis 不存在时）：
@Bean
public CacheManager cacheManager() {
    return new NoOpCacheManager();  // @Cacheable 注解不生效，不报错
}

// 业务代码（现在就写）：
@Override
@Cacheable(value = "services", key = "'all'")
public List<ServiceItem> list() { ... }

// 将来 Redis 就位后：删掉 NoOpCacheManager，换 RedisCacheManager
// 所有 @Cacheable 立即生效，零业务代码改动
```

**模式 2：事件预留 — `ApplicationEvent` + `@EventListener`**

```java
// 当前（同步，不需要 MQ）：
@Transactional
public void createOrder(CreateOrderDTO dto) {
    // ... 核心写入 ...
    eventPublisher.publishEvent(new OrderCreatedEvent(order.getId(), ...));
}

@EventListener
public void onOrderCreated(OrderCreatedEvent event) {
    // 积分发放、会员升级 — 当前同步执行
    pointsService.addPoints(event.memberId(), calculatePoints(event));
}

// 将来 MQ 就位后：
// 改造只有 1 行：eventPublisher.publishEvent(...) → rabbitTemplate.convertAndSend(...)
// @EventListener 改为 @RabbitListener，方法体复用
```

**模式 3：配置外置 — `${ENV_VAR}` 区分环境**

```yaml
# application-dev.yml — 本地直连，无中间件
spring:
  redis:
    host: ${REDIS_HOST:localhost}    # 本地没 Redis 时启动不会报错（spring-boot-starter-data-redis 未引入）

# application-prod.yml — 必须配置，缺失启动即失败
spring:
  redis:
    host: ${REDIS_HOST}              # 无默认值，防止遗漏配置
    password: ${REDIS_PASSWORD}
```

### 8.5 上线前固化的执行顺序

见 [docs/12-商用代码审计未修复项-2026-05-22.md](docs/12-商用代码审计未修复项-2026-05-22.md) §十，汇总如下：

| 阶段 | 内容 | 时间 |
|------|------|------|
| Phase 9 | Redis（分布式锁 + 限流 + 会话 + 缓存 + 幂等） | 3-5天 |
| Phase 10 | 连接池调优 + 线程池治理 + 优雅关闭 | 半天（✅ 已完成） |
| Phase 11 | RabbitMQ（订单/支付/短信异步） | 3-5天 |
| Phase 12 | Prometheus + 健康检查 + 结构化日志 | 1天 |
| Phase 13 | 多实例部署 + Nginx 负载均衡 | 1天 |

### 8.6 关键原则

1. **中间件是放大器，不是替代品。** 如果代码本身有并发Bug（read-modify-write），加 Redis 只是让 Bug 更快发生
2. **能在代码层解决的问题，不用中间件。** 原子 SQL（`WHERE status=old`）和乐观锁（`rows==0`）能处理的并发，不引入分布式锁
3. **Docker Compose 管理中间件生命周期。** 开发时不启动，需要时一行命令：`docker compose -f docker-compose.dev.yml up -d`
4. **代码结构先对齐，中间件后接入。** 现在的 `@Cacheable`、`ApplicationEvent`、`${ENV_VAR}` 就是为将来准备好的接口

## 九、技术债管理规则

为避免再次积累，从 Phase 5 开始执行以下规则：

1. **每新增一个 API 端点，必须带至少 1 个集成测试**
2. **每新增一个 Entity，必须检查 DTO/VO 是否已定义**（禁止 Entity 直接暴露）
3. **敏感配置一律用环境变量，禁止硬编码**
4. **每完成一个模块，更新对应的 docs 文件**（04/05/06/07/13/16）
5. **PR/提交前执行 mvn compile + npm run build，两者均通过才合并**
